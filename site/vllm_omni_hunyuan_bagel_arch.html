vLLM-Omni：Hunyuan-Image3.0 与 BAGEL 架构走读

本文只基于当前仓库里的 vllm-omni 源码整理，不引入外部资料。目标是先把 Hunyuan-Image3.0 和 BAGEL 的模型组件及阶段串起来，再抽象出 vllm-omni 对这类“AR/Thinker + DiT/VAE”模型复用的公共框架。

1. 背景：为什么这两类图像模型都适合 vllm-omni

Hunyuan-Image3.0 和 BAGEL 都不是“一个 diffusion pipeline 从 prompt 直接到图片”的简单形态。它们都需要一个语言/多模态前端先把文本、图片、推理链或条件上下文组织好，再交给图像生成后端：

模型前端 stage后端 stage跨 stage 交付物vllm-omni 复用点
Hunyuan-Image3.0 AR，HunyuanImage3ForCausalMM dit，HunyuanImage3Pipeline AR 文本中的 </recaption> 前缀、<img_ratio_*> token、原始条件图、可选 AR KV custom_process_input_func=ar2diffusion，KV connector，diffusion runner
BAGEL-7B-MoT thinker，OmniBagelForConditionalGeneration dit，BagelPipeline 主请求 KV、CFG companion 请求 KV、CFG role 元数据 prompt_expand_func，cfg_kv_collect_func，KV connector

所以 vllm-omni 的价值不只是“服务 diffusion 模型”，而是把多阶段模型拆成可调度、可部署、可跨设备传输的 stage：

OpenAI-compatible API→ AsyncOmniEngine→ Stage 0: LLM_AR→ model-specific bridge→ Stage 1: DIFFUSION→ image/text output

2. Hunyuan-Image3.0：AR 产出 CoT / ratio，DiT 负责去噪成图

2.1 Pipeline 拓扑

Hunyuan-Image3.0 的默认 pipeline 在 vllm_omni/model_executor/models/hunyuan_image3/pipeline.py，model_type 是 hunyuan_image_3_moe。它声明两个 stage：

HUNYUAN_IMAGE3_PIPELINE = PipelineConfig(
    model_type="hunyuan_image_3_moe",
    stages=(
        StagePipelineConfig(
            stage_id=0,
            model_stage="AR",
            execution_type=StageExecutionType.LLM_AR,
            input_sources=(),
            final_output=False,
            engine_output_type="latent",
        ),
        StagePipelineConfig(
            stage_id=1,
            model_stage="dit",
            execution_type=StageExecutionType.DIFFUSION,
            input_sources=(0,),
            final_output=True,
            final_output_type="image",
            custom_process_input_func=(
                "vllm_omni.model_executor.stage_input_processors."
                "hunyuan_image3.ar2diffusion"
            ),
        ),
    ),
)

部署 YAML vllm_omni/deploy/hunyuan_image_3_moe.yaml 把 stage 0 放在 devices: "0,1"，stage 1 放在 devices: "2,3"，并打开 omni_kv_config.need_send_cache / need_recv_cache。这说明 Hunyuan 的跨 stage 信息分两条路：一条是显式 prompt bridge，另一条是 KV cache connector。

AR / tokenizer / 文本链路 条件图像编码 DiT / VAE 成图 vllm-omni connector

2.2 Hunyuan bridge 的关键语义

ar2diffusion 不只是把 AR 文本复制给 DiT，它还修正目标尺寸，并把图片条件重新挂到 diffusion prompt 上：

ratio_idx = _extract_ratio_index(generated_token_ids)
if ratio_idx is not None:
    base_size = int(original_prompt.get("image_base_size", 1024))
    size_table = _build_ratio_size_table(base_size)
    if 0 <= ratio_idx < len(size_table):
        height, width = size_table[ratio_idx]

cot_text_for_dit = _truncate_at_cot_end(generated_text)

diffusion_input = {
    "prompt": text_prompt,
    "height": height,
    "width": width,
    "extra": {"ar_generated_text": cot_text_for_dit},
}
理解 Hunyuan 的核心：AR stage 负责“想清楚要生成什么”和“决定图像比例”，DiT stage 负责“把这个文本/比例/条件图像变成 latent denoising 轨迹”。如果没有 ar2diffusion 的 ratio token 解析，image editing 场景很容易退回 serving 层携带的默认宽高。

2.3 Hunyuan 模型内部组件图

从模型本身看，Hunyuan-Image3.0 有两套很像的多模态组件：AR stage 用它们把输入图像变成可插入 LLM 的 embedding；DiT stage 再用同类组件把条件图和生成 latent 一起送进 diffusion transformer。

组件在 AR stage 的作用在 DiT stage 的作用
AutoencoderKLConv3D / DistributedAutoencoderKLHunyuan 把输入图像编码成 VAE latent，再经 UNetDown 投到 hidden size，作为图片 token 插入 LLM 上下文。 条件图仍然走 VAE encode；生成结束后用 VAE decode 把最终 latent 还原成图。
Siglip2VisionTransformer + LightProjector 提供语义视觉 token，和 VAE token 之间用 <joint_img_sep> 隔开。 image editing / multi-image conditioning 时继续提供条件图语义特征。
HunyuanModel 标准 AR decoder，带 mRoPE；采样器会强制 </think> → <recaption> 等阶段 token，并限制 ratio token。 同一类 transformer 作为 DiT 主体，输入是文本/条件图 token + 当前噪声 latent token，输出 diffusion prediction。
UNetDown / UNetUp UNetDown 把 VAE latent patch 化成 token embedding。 UNetDown 处理当前噪声 latent，UNetUp final_layer 把 hidden states 转回 latent patch prediction。

3. BAGEL：Thinker 产出 KV，DiT 用三路 CFG 分支成图

3.1 Pipeline 拓扑

BAGEL 的默认 pipeline 在 vllm_omni/model_executor/models/bagel/pipeline.py。它也是两阶段，但不是通过 custom_process_input_func 改写 prompt，而是通过 KV cache 让 DiT 直接复用 Thinker 的上下文。

BAGEL_PIPELINE = PipelineConfig(
    model_type="bagel",
    model_arch="OmniBagelForConditionalGeneration",
    stages=(
        StagePipelineConfig(
            stage_id=0,
            model_stage="thinker",
            execution_type=StageExecutionType.LLM_AR,
            prompt_expand_func="...bagel.expand_cfg_prompts",
            omni_kv_config={
                "need_send_cache": True,
                "kv_transfer_criteria": {"type": "prefill_finished"},
            },
        ),
        StagePipelineConfig(
            stage_id=1,
            model_stage="dit",
            execution_type=StageExecutionType.DIFFUSION,
            input_sources=(0,),
            cfg_kv_collect_func="...bagel.collect_cfg_kv_caches",
            omni_kv_config={"need_recv_cache": True},
        ),
    ),
)

vllm_omni/deploy/bagel.yaml 默认让 Thinker 和 DiT 同在 devices: "0"，stage 之间用 SharedMemoryConnector。多机或 RDMA 场景可以换 MooncakeTransferEngineConnector。

3.2 BAGEL 的 CFG request expansion

BAGEL 的 expand_cfg_prompts 会根据输出模态和是否 img2img 生成 companion request。核心规则：

场景主请求额外请求DiT 侧使用
text2img，有 negative_prompt gen 正向 prompt cfg_text negative prompt cfg_img 复用主请求 KV
img2img 图片 + edit prompt cfg_text、cfg_img 三路 KV 分别作为 image/text CFG 条件
text2text / img2text 正常理解或文本输出 无 不走图像 CFG 
def expand_cfg_prompts(prompt, sampling_params):
    modalities = prompt.get("modalities", [])
    if "image" not in modalities and "img2img" not in modalities:
        return []

    neg_prompt = _get_negative_prompt(prompt, sampling_params)

    if "image" in modalities:
        if not neg_prompt:
            return []
        return [ExpandedPrompt(
            prompt={"prompt": neg_prompt, "modalities": prompt.get("modalities", [])},
            role="cfg_text",
            request_id_suffix="__cfg_text",
        )]

	    if "img2img" in modalities:
	        return [
	            ExpandedPrompt(... role="cfg_text", request_id_suffix="__cfg_text"),
	            ExpandedPrompt(... role="cfg_img", request_id_suffix="__cfg_img"),
	        ]

3.3 BAGEL 模型内部组件图

BAGEL 的名字里 MoT 是重点：同一个 Qwen2MoT backbone 里，普通文本/ViT token 和生成用 VAE latent token 走不同的投影、norm、MLP 分支。vllm-omni 的两阶段部署只是把“前缀上下文构建”和“latent 去噪”拆开，模型内部仍然是这套 MoT 组件在工作。

组件具体作用为什么对理解 BAGEL 重要
Qwen2MoTForCausalLM 主干语言模型，内部 decoder layer 有普通理解分支和 *_moe_gen 生成分支。 它不是单纯 LLM：VAE latent token 会走生成专用 QKV / norm / MLP，文本和 ViT token 走理解分支。
vae2llm / llm2vae vae2llm 把 VAE latent patch 投到 hidden size；llm2vae 把 hidden state 投回 latent patch 维度。 这是 LLM token 空间和图像 latent 空间之间的真正桥梁。
SiglipNaViTWrapper + MLPconnector 把输入图片变成理解用视觉 token，再投到 Qwen hidden size。 img2img 或图像理解时，它提供语义条件；VAE 分支提供像素/latent 条件。
generate_image 从 packed_init_noises 开始按 timestep 更新 latent，主分支、cfg_text、cfg_img 分支共同决定速度预测。 BAGEL 的 CFG 不是简单 negative prompt 字符串，而是三套 KV context 对同一批 latent query 的条件控制。

4. vllm-omni 只负责把模型组件接起来

框架部分可以压到很小：vllm-omni 不改变 Hunyuan 或 BAGEL 的内部计算图，只负责把“前端上下文构建”和“后端图像生成”拆成 stage，并在 stage 边界调用少量模型专用 hook。

框架接口Hunyuan-Image3.0 用法BAGEL 用法
PipelineConfig 声明 AR → dit，下游用 ar2diffusion 组装 DiT prompt。 声明 thinker → dit，下游主要读取 Thinker KV。
prompt_expand_func 不需要。AR 自己生成 recaption / ratio。 生成 cfg_text / cfg_img companion request。
custom_process_input_func ar2diffusion：截断 CoT、解析 ratio token、转交条件图。 默认不用，DiT 从 KV context 和原 prompt 取条件。
cfg_kv_collect_func 不需要三路 CFG companion KV。 collect_cfg_kv_caches：把 companion KV 挂成 cfg_text_past_key_values / cfg_img_past_key_values。

4.1 最小拓扑：PipelineConfig 是模型结构，DeployConfig 是部署结构

StagePipelineConfig 里只放模型结构：stage id、execution type、输入来源、是否最终输出、模型专用 hook。设备、并行、connector、默认 sampling 参数才放 deploy YAML。

@dataclass(frozen=True)
class StagePipelineConfig:
    stage_id: int
    model_stage: str
    execution_type: StageExecutionType = StageExecutionType.LLM_AR
    input_sources: tuple[int, ...] = ()
    final_output: bool = False
    final_output_type: str | None = None
    custom_process_input_func: str | None = None
    prompt_expand_func: str | None = None
    cfg_kv_collect_func: str | None = None
    omni_kv_config: dict[str, Any] | None = None

4.2 两个模型真正依赖的框架动作

对于 Hunyuan 这类“AR 输出需要改写成 diffusion prompt”的模型，关键路径在 orchestrator.py：

if next_pool.stage_type == "diffusion":
    diffusion_source_outputs = [output, *companion_outputs]
    if next_client.custom_process_input_func is not None:
        diffusion_prompt = next_client.custom_process_input_func(
            diffusion_source_outputs,
            req_state.prompt,
            requires_multimodal_data,
            **_extra_kwargs,
        )
    else:
        diffusion_prompt = req_state.prompt

    await next_pool.submit_initial(
        req_id,
        req_state,
        diffusion_prompt,
        submit_kwargs={"kv_sender_info": ...},
        params_override=self._maybe_clone_diffusion_params_for_cfg(req_id, params),
    )

对于 BAGEL，custom_process_input_func 为空，但 prompt_expand_func 会在主请求入队后额外创建 companion request：

if self.prompt_expand_func is not None and final_stage_id > 0:
    original_prompt = msg.original_prompt
    effective_spl = msg.sampling_params_list
    stage0_params = effective_spl[0]
    self._enqueue_cfg_companions(request_id, original_prompt, stage0_params, effective_spl)

4.3 KV 只作为模型上下文的一种载体

Diffusion runner 每次执行前都会尝试收 KV，是否真的收由 need_recv_cache 决定。BAGEL 的 CFG companion KV 也是在这里被收集并挂到 req.sampling_params 上：

self.kv_transfer_manager.receive_multi_kv_cache_distributed(
    req,
    cfg_kv_collect_func=getattr(self.od_config, "cfg_kv_collect_func", None),
    target_device=getattr(self.pipeline, "device", None),
)

5.1 Hunyuan text-to-image

请求形态

POST /v1/images/generations
{
  "model": "tencent/HunyuanImage-3.0-Instruct",
  "prompt": "A glass teapot on a bamboo table",
  "width": 1024,
  "height": 1024,
  "n": 1,
  "num_inference_steps": 50,
  "guidance_scale": 0
}

阶段变量

节点核心变量
stage 0 ARoutput.text，output.cumulative_token_ids
ar2diffusionratio_idx，cot_text_for_dit，diffusion_input
stage 1 DiTcot_text，image_size，ar_kv_data
输出DiffusionOutput(output=outputs[0])
  1. AsyncOmniEngine 根据 pipeline 生成两个 stage 的 sampling_params_list，并把请求提交给 stage 0。
  2. stage 0 AR 生成文本和特殊 token。对 Hunyuan 来说，尾部的 <img_ratio_*> 是目标比例的机器可读信号。
  3. Orchestrator 发现下游是 diffusion stage，调用 hunyuan_image3.ar2diffusion(source_outputs, original_prompt, ...)。
  4. ar2diffusion 从 token id 反查 ratio index，用 _build_ratio_size_table(base_size) 得到 (height,width)，再截断 </recaption> 后面的尾部 token。
  5. HunyuanImage3Pipeline.forward 读取 extra.ar_generated_text，归一化成 cot_text，调用 prepare_model_inputs(... mode="gen_image")。
  6. prepare_model_inputs 构建 chat template、2D RoPE、条件图像编码、CFG batch，然后 _generate 执行 FlowMatch denoising，最后 VAE decode。

请求形态

{

  "modalities": ["image"],
  "extra_body": {
    "negative_prompt": "blurry, low quality",
    "num_inference_steps": 50,
    "cfg_text_scale": 4.0,
    "cfg_img_scale": 1.5
  }
}

阶段变量

节点核心变量
prompt_expand_funcExpandedPrompt(role="cfg_text")
stage 0 Thinkerrequest_id 与 request_id__cfg_text 的 KV
KV collectcfg_text_past_key_values，cfg_text_kv_metadata
stage 1 DiTgen_context，cfg_text_context，cfg_img_context
  1. 主请求入队后，AsyncOmniEngine._enqueue_cfg_companions 调用 expand_cfg_prompts，创建 request_id__cfg_text。
  2. 主请求和 companion 请求都经过 stage 0 prefill。由于 kv_transfer_criteria={"type":"prefill_finished"}，prefill 完成即可发送 KV。
  3. stage 1 的 DiffusionModelRunner.execute_model 先调用 receive_multi_kv_cache_distributed，主 KV 被挂到 sampling_params.past_key_values。
  4. collect_cfg_kv_caches 再按 role 拉取 companion KV，并设置 cfg_text_past_key_values 等字段。
  5. BagelPipeline.forward 将主 KV 填入 gen_context，将 companion KV 填入 cfg_text_context；text2img 下 cfg_img_context 默认复用主 KV。
  6. bagel.generate_image 同时使用三路 context 做 CFG，得到 latent，再由 AutoEncoder.decode 转成 PIL.Image。

6.2 新增类似模型时的最小接口

1. 先声明拓扑

在 model_executor/models/<model>/pipeline.py 定义 PipelineConfig。如果是 AR→DiT，就让 stage 1 的 input_sources=(0,)，并声明需要的 hook。

2. 再声明 bridge

如果下游 prompt 需要从上游输出构造，写 custom_process_input_func。如果需要 CFG companion，就写 prompt_expand_func 和 cfg_kv_collect_func。

3. 最后补 deploy

在 deploy YAML 中配置每个 stage 的 devices、并行参数、默认 sampling 参数和 connector。结构不要塞进 deploy，结构应该在 pipeline 里。

一个常见误区：custom_process_input_func 和 KV connector 不是二选一。Hunyuan 同时用显式 prompt bridge 和 KV reuse；BAGEL 主要靠 KV bridge，但也用 prompt_expand_func 在请求层制造 CFG 分支。

7. 总结

  • Hunyuan-Image3.0 的主线是 AR 文本/ratio/cot → ar2diffusion → DiT/VAE。它的 bridge 要同时处理文本截断、比例 token、系统 prompt、原始条件图和可选 AR KV。
  • BAGEL 的主线是 Thinker prefill KV → CFG companion KV → BagelPipeline 三路 CFG。它的关键不是改写 prompt，而是保证主请求和 companion 请求的 KV 被准确收集。
  • vllm-omni 的公共框架把模型差异压缩到四个 hook：prompt_expand_func、custom_process_input_func、custom_process_next_stage_input_func、cfg_kv_collect_func。
  • PipelineConfig 定义“模型是什么结构”，deploy YAML 定义“这个结构怎么部署”。这条边界清楚后，Hunyuan、BAGEL、Qwen-Omni、GLM-Image 等模型可以复用同一套 stage 编排和 connector。
