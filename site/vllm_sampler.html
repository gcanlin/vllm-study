vLLM Sampler 过程导读

本文按当前仓库代码梳理 vLLM v1 的采样路径：从请求的 SamplingParams 进入 GPU runner，到 logits 被约束、惩罚、temperature/top-k/top-p/min-p 处理，再到 Gumbel-max 选出 token 并回传给调度器。

主线代码：vllm/vllm/v1/worker/gpu/model_runner.py、vllm/vllm/v1/worker/gpu/sample/。兼容路径：vllm/vllm/v1/sample/sampler.py。

1. 两套 Sampler 路径

当前仓库里能看到两套 v1 sampler。理解时先分清楚入口，否则很容易把两个实现的顺序混在一起。

路径入口特点适合怎么看
vllm/v1/worker/gpu/sample/Sampler worker/gpu/model_runner.py stateful：请求参数常驻 RequestState / UvaBackedTensor / StagedWriteTensor；采样使用 Triton Gumbel-max；按 InputBatch 的 req-state 映射处理。 主线。本文 2-8 节按这条讲。
vllm/v1/sample/Sampler gpu_model_runner.py、部分 warmup / rejection sampler 场景 stateless-ish：每步传入 SamplingMetadata；随机采样由 TopKTopPSampler 选择 FlashInfer、Triton、CPU/XPU/native 路径。 第 9 节单独对照。
一句话：新 GPU worker 把 per-request sampling state 长期放在 GPU/UVA 侧，采样时只拿本 step 的 idx_mapping 做索引；兼容路径每步围绕 SamplingMetadata 组织采样。

2. 端到端主流程

主流程在 worker/gpu/model_runner.py 里非常直接：最后一个 PP rank 取要产出 logits 的 hidden states，调用模型的 compute_logits()，应用结构化输出 grammar bitmask，然后进入 sampler。如果这一批包含 draft tokens，则走 rejection sampler。

sample_hidden_states = hidden_states[input_batch.logits_indices]
logits = self.model.compute_logits(sample_hidden_states)
if grammar_output is not None:
    structured_outputs_worker.apply_grammar_bitmask(...)

if input_batch.num_draft_tokens == 0:
    sampler_output = self.sampler(logits, input_batch)
else:
    sampler_output = self.rejection_sampler(logits, input_batch, draft_logits)

3. SamplingParams 如何变成 GPU 状态

新请求进入 GPU runner 时，sampler.add_request(req_index, prompt_len, sampling_params) 会把请求参数拆到多个状态对象中。后续每个 step 不再重新构造一份大 metadata，而是通过 idx_mapping 找到每个 batch row 对应的 request state。

SamplingStates

保存 temperature、top_k、top_p、min_p、seed、num_logprobs。其中 top_k<=0 或超过 vocab 时会被归一化为 vocab_size，表示不启用 top-k。

UvaBackedTensorper request

LogitBiasState

保存 allowed_token_ids、logit_bias、min_tokens 对应的 stop token mask。它们属于“会影响 greedy 结果”的处理。

allowed idsstop token

PenaltiesState

保存 repetition / frequency / presence penalty，并维护 prompt token bitset 和 output token 计数。这样采样时 kernel 可以直接知道某 token 是否在 prompt/output 中出现过。

prompt_bin_maskoutput_bin_counts

BadWordsState / LogprobTokenIdsState

bad words 被扁平化成 token 数组和 offsets；指定 token 的 logprob 请求也被单独缓存，避免每步临时整理复杂 Python 结构。

staged writeGPU gather
关键点：apply_staged_writes() 把 CPU 侧 staged 参数同步到 GPU/UVA 侧。之后采样阶段只处理当前 batch 中被 idx_mapping_np 选中的 request。

4. 从 hidden states 到 logits

InputBatch 里和 sampler 相关的字段主要有：

字段shape / 含义采样中怎么用
logits_indices本 step 需要算 logits 的 token 位置hidden_states[input_batch.logits_indices] 只取最后要预测的位置；prefill 可能不是每个输入 token 都产出采样 token。
idx_mapping / idx_mapping_npbatch row 到 request state index从 state tensors 中取对应请求的 temperature、top-k、penalty 等。
expanded_idx_mappingspec decoding 时可能比 request 数更长多个 draft/logit row 映射回同一个 request state。
positions每个 token 在 request 内的位置Gumbel seed 用 seed + pos 派生随机性；min_tokens 也依赖 position。
expanded_local_pos当前 logit row 在该 request 的本 step 内局部位置penalty 和 bad words 在 spec token 场景下需要看同 step 内前面的 draft token。

所以 sampler 看到的 logits 通常是 [num_logits, vocab_size]，不是完整 [num_input_tokens, vocab_size]。decode 常见是一请求一行；prefill、chunked prefill、spec decode 会让行数和请求数不完全一致。

5. logits 处理顺序

主线 sampler 的 apply_sampling_params() 顺序如下。多数操作是 in-place kernel，目标是尽量少分配、少 CPU/GPU 同步。

5.1 logit bias 阶段

LogitBiasState.apply_logit_bias() 的 Triton kernel 同时处理三类约束：

  • allowed_token_ids：先把整行 logits 置为 -inf，再恢复白名单 token 的原 logits。
  • logit_bias：对指定 token 加 bias。
  • min_tokens：当前位置未达到最小生成长度时，把 stop token logits 置为 -inf。

5.2 penalties 阶段

PenaltiesState 预先把 prompt 中出现过的 token 做成 bit mask，把已经输出过的 token 做成 counts。采样时 kernel 对每个 vocab block 做：

if token appeared in prompt or output: repetition penalty: logits > 0: logits /= repetition_penalty logits <=0: logits *= repetition_penalty frequency penalty: logits -= frequency_penalty * output_count[token] presence penalty: logits -= presence_penalty * (output_count[token] > 0)

5.3 bad words、temperature、min-p、top-k/top-p

BadWordsState 检查 bad word 的 prefix 是否和已输出 token 及本 step 里的 spec input 匹配，匹配时把 bad word 的最后一个 token 置为 -inf。之后 temperature kernel 对每行按请求温度除法；min_p 以本行最大 logit 为基准，低于 max_logit + log(min_p) 的 token 被屏蔽；top-k/top-p 最后再把候选集合裁小。

主线 GPU sampler 里 temperature 在 top-k/top-p 之前完成；temperature == 0 会跳过随机噪声，后面的 Gumbel-max 退化成 argmax。

6. Gumbel-max 采样

主线 sampler 不调用 torch.multinomial。它用 Gumbel-max trick：对每个 token 的处理后 logit 加一份 Gumbel 噪声，然后取 argmax。

sample = argmax_i(logit_i + g_i) g_i = -log(-log(u_i)), u_i ~ Uniform(0, 1)

当 temperature 为 0 时，kernel 不加 Gumbel 噪声，直接在处理后的 logits 上取最大值，也就是 greedy。随机种子来自请求的 seed，再结合当前位置 pos 派生，因此同一请求不同 token 位置不会复用同一随机序列。

步骤实现细节
分块扫描 vocab_gumbel_sample_kernel按 BLOCK_SIZE=1024 对 vocab 分块，每块得到局部最大值和 token id。
局部结果归并local_max.argmax() + local_argmax.gather()第二阶段在 PyTorch 中从每行所有 block 的局部最大值里选全局最大。
fp64 可选use_fp64_gumbel默认 fp32；开启后 Gumbel 噪声计算更精细但吞吐更低。
processed_logits = self.apply_sampling_params(...)
sampled = gumbel_sample(
    processed_logits,
    expanded_idx_mapping,
    self.sampling_states.temperature.gpu,
    self.sampling_states.seeds.gpu,
    pos,
    apply_temperature=False,
)

7. logprobs 回收

采样结束后，sampler 会看本 batch 是否有人请求 logprobs 或 logprob_token_ids。如果没有，直接返回 token，避免额外 top-k / log-softmax 成本。

有 logprobs 时，主线代码走 compute_topk_logprobs()，输入可以是原始 logits，也可以是处理后的 logits，取决于 logprobs_mode。输出会放进 SamplerOutput.logprobs_tensors，随后在 async output copy 阶段转到 CPU，最终交给 engine output processor。

字段作用
sampled_token_idsGPU tensor，shape 通常是 [num_reqs, 1]；spec decoding 可返回多 token。
logprobs_tensors包含 sampled token 以及 top-k / 指定 token 的 logprob 和 rank。
num_sampled每个请求这步真正接受/生成了多少 token；chunked prefill 可能是 0。
num_nans仅在 VLLM_COMPUTE_NANS_IN_LOGITS 开启时统计。

8. 投机解码分支

当 input_batch.num_draft_tokens != 0 时，model_runner.sample() 不直接调用普通 sampler，而是调用 RejectionSampler。直觉上它要做两件事：

  1. 对 target model 的 logits 应用同一套采样约束。
  2. 根据 draft logits 和 target logits 接受或拒绝 draft token，必要时从 target 分布补采样。

普通 sampler 仍然是它的基础组件，因为 draft 被拒绝后的补 token、logprobs 以及部分 bonus token 预测都需要复用同一套 logits 处理和采样语义。主线里 expanded_idx_mapping、expanded_local_pos、bad words 和 penalties 对 spec token 的处理，都是为了让一个 step 内多 token 采样也保持一致。

9. 兼容 Sampler 路径对照

vllm/v1/sample/sampler.py 的类注释已经把顺序写得很完整。它的核心差异是：每次 forward(logits, SamplingMetadata) 带着本 step 的完整 metadata 进来，sample() 中先尝试 greedy，再按 temperature / min-p / top-k/top-p 做随机采样。

forward 顺序

  1. 必要时先用原始 logits 计算 raw logprobs。
  2. logits 转 fp32。
  3. allowed token、bad words、非 argmax-invariant logits processor。
  4. repetition / frequency / presence penalties。
  5. sample() 执行 greedy 或 random。
  6. 按需 gather top logprobs / 指定 token logprobs。

TopKTopPSampler

它会根据平台和模式选择实现：CUDA 上可用 FlashInfer；CPU 走 CPU/native；XPU 走专用 kernel；ROCm 有 aiter 分支但当前代码里采样 path 仍禁用回退。native 路径会先把 top-k/top-p 外的 logits 置 -inf，再 softmax。

随机采样方式

兼容路径的 native random sample 使用 exponential race，而不是 torch.multinomial：

q_i ~ Exponential(1) sample = argmax_i(prob_i / q_i)

这和按概率分布采样在统计上等价，并避免了 torch.multinomial 带来的 CPU-GPU 同步。FlashInfer 路径也声明只保证统计等价，不保证和 native 输出逐 token 一致。

10. 调试核对清单

想确认的问题看哪里预期
本请求的 temperature/top-k/top-p 是否已进入 GPU 状态？worker/gpu/sample/states.py::SamplingStates.add_request/apply_staged_writestop_k<=0 被改成 vocab_size；seed 为 None 时生成随机 int64。
为什么 stop token 没出来？worker/gpu/sample/logit_bias.py::_bias_kernel如果 pos < min_len，stop token 被置 -inf。
为什么只有白名单 token 能采？LogitBiasStateallowed ids 会先清空整行 logits，再恢复白名单。

为什么 temperature=0 仍走 gumbel_sample？worker/gpu/sample/gumbel.py::gumbel_block_argmaxtemp == 0 时不加 Gumbel 噪声，结果是 greedy argmax。
top-p/top-k 小 batch 为什么不是 Triton？v1/sample/ops/topk_topp_sampler.py::apply_top_k_top_p非 CPU 且 batch size 大于等于 8 才优先 Triton，否则用 PyTorch sort/topk 路径。
