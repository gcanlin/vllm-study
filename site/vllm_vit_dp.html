vLLM ViT DP 代码导读：以 Qwen-VL 为例

本文讲的是 --mm-encoder-tp-mode data：在一个 TP 组内，每张卡都持有完整视觉 encoder 权重，但把图片/视频 item 分给不同 TP rank 并行编码，再 all-gather 回完整 embedding。

代码基于当前工作区 vllm/，引用均给出 文件:行号。Qwen2-VL、Qwen2.5-VL、Qwen3-VL 的主路径一致，文中用 Qwen3-VL 展开 CUDA Graph，用 Qwen2-VL 展开 eager 路径。

1. ViT DP 是什么

vLLM 里多模态视觉 encoder 默认跟随 TP 做权重切分，也就是 mm_encoder_tp_mode="weights"。ViT DP 把这个策略改成 "data"：TP 组内的每个 rank 保留完整视觉 encoder 权重，输入 batch 里的图片/视频按 item 粒度切给不同 rank，本地编码后用 TP 通信组 all-gather，再恢复原始 item 顺序。

这不是服务层请求级 DP。--data-parallel-size 是多个 engine replica 分摊请求；--mm-encoder-tp-mode data 是同一个 engine 的 TP rank 在视觉 encoder 阶段临时当作数据并行 worker。
模式视觉 encoder 权重视觉输入适合场景
weights跟 TP 一样切权重每个 rank 处理同一批输入的分片计算默认模式，省单卡权重显存
data每个 TP rank 一份完整权重图片/视频 item 按负载分配到 rank多图、多视频、动态分辨率，ViT 计算成为瓶颈

配置对象直接把这个含义写在注释里：MultiModalConfig.mm_encoder_tp_mode 的 "data" 表示“split the batched input data across TP ranks”，同时说明它不是 --data-parallel-size 控制的请求级 DP（vllm/vllm/config/multimodal.py:145-155）。

2. 如何使用

2.1 在线服务

vllm serve Qwen/Qwen2.5-VL-7B-Instruct \
  --tensor-parallel-size 4 \
  --mm-encoder-tp-mode data \
  --limit-mm-per-prompt '{"image": 8, "video": 1}'

--mm-encoder-tp-mode 是 CLI 参数，注册点在 vllm/vllm/engine/arg_utils.py:1233-1238。实际使用时需要 --tensor-parallel-size > 1 才有可并行的 TP rank；TP=1 时选择 data 也没有并行收益。

2.2 离线 LLM

from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen2.5-VL-7B-Instruct",
    tensor_parallel_size=4,
    mm_encoder_tp_mode="data",
    limit_mm_per_prompt={"image": 8},
)

outputs = llm.generate(
    {
        "prompt": "<|vision_start|><|image_pad|><|vision_end|>描述这张图。",
        "multi_modal_data": {"image": image},
    },
    SamplingParams(max_tokens=128),
)

2.3 判断模型是否支持

模型接口有显式能力位 supports_encoder_tp_data，默认是 False（vllm/vllm/model_executor/models/interfaces.py:113-117）。Qwen2-VL 和 Qwen3-VL 都设置为 True（qwen2_vl.py:1198，qwen3_vl.py:1626）。不支持的模型即使传了 data，也不应该假设它会走 DP 代码。

3. 整体框架图

关键点：ViT DP 只包住视觉 encoder 的执行；LLM decode/prefill 仍按原来的 TP/PP/attention 路径走。

4. Qwen-VL 模型入口

Qwen2-VL 的入口最短，适合看清楚分支：

supports_encoder_tp_data = True

def __init__(...):
    multimodal_config = vllm_config.model_config.multimodal_config
    self.use_data_parallel = multimodal_config.mm_encoder_tp_mode == "data"

def _process_image_input(self, image_input):
    grid_thw = image_input["image_grid_thw"]
    if image_input["type"] == "image_embeds":
        image_embeds = image_input["image_embeds"]
    else:
        pixel_values = image_input["pixel_values"]
        if self.use_data_parallel:
            return run_dp_sharded_mrope_vision_model(
                self.visual, pixel_values, grid_thw.tolist(), rope_type="rope_3d"
            )
        else:
            image_embeds = self.visual(pixel_values, grid_thw=grid_thw)

    merge_size = self.visual.spatial_merge_size
    sizes = (grid_thw.prod(-1) // merge_size // merge_size).tolist()
    return image_embeds.split(sizes)

来源：vllm/vllm/model_executor/models/qwen2_vl.py:1198、1290-1293、1363-1384。

视频路径同理，只是读 pixel_values_videos 和 video_grid_thw，同样调用 run_dp_sharded_mrope_vision_model(..., rope_type="rope_3d")（qwen2_vl.py:1386-1409）。

Qwen3-VL 也设置 supports_encoder_tp_data = True，在构造函数中用同一行配置开关 self.use_data_parallel = multimodal_config.mm_encoder_tp_mode == "data"（qwen3_vl.py:1626、1646-1657）。它额外实现了 SupportsEncoderCudaGraph，所以除了 eager 分支外，还有 CUDA Graph fast path。

5. Eager DP 代码走读

核心函数在 vllm/vllm/model_executor/models/vision.py:383-570。它的输入是已经被 processor 展平后的 pixel_values，以及每个 item 的 grid_thw。

5.1 先按 item 算负载

tp_size = get_tensor_model_parallel_world_size()
tp_rank_local = get_tensor_model_parallel_rank()

patches_per_image = [math.prod(grid_thw) for grid_thw in grid_thw_list]
cum_patches_per_image = [0, *itertools.accumulate(patches_per_image)]

Qwen-VL 的 pixel_values 是所有图片/视频 patch 在第 0 维拼起来的长 tensor；grid_thw 给出每个 item 的 patch 数。比如 4 张图的 patch 数是 [1000, 100, 200, 50]，累加偏移就是 [0, 1000, 1100, 1300, 1350]。

5.2 贪心负载均衡

large_to_small_indices = sorted(
    range(n_samples), key=lambda i: sizes[i], reverse=True
)

for idx in large_to_small_indices:
    min_gpu = min(range(num_gpus), key=lambda i: gpu_loads[i])
    gpu_assignments[min_gpu].append(idx)
    gpu_loads[min_gpu] += sizes[idx]

实现是按 patch 数从大到小排，每次把最大剩余 item 放到当前负载最小的 rank（vision.py:314-380）。这比简单 round-robin 更适合动态分辨率图片，因为大图和小图的 ViT 计算差异很大。

5.3 每个 TP rank 切本地输入

image_idxs_local = image_to_tp_rank[
    cum_gpu_sample_counts[tp_rank_local] : cum_gpu_sample_counts[tp_rank_local + 1]
]

pixel_values_local = torch.cat([
    pixel_values[cum_patches_per_image[i] : cum_patches_per_image[i + 1]]
    for i in image_idxs_local
])

这里没有切单张图的 patch，而是按完整 item 切。这样 grid_thw、MRoPE、后续 split 都保持简单；代价是如果只有一张超大图，TP 组内仍然只有一个 rank 真正在跑 ViT。

5.4 本地跑完整视觉模型

if rope_type == "rope_3d":
    image_embeds_local = vision_model(pixel_values_local, local_grid_thw_list)
else:
    image_embeds_local = vision_model(
        pixel_values_local, torch.tensor(local_grid_thw_list)
    )

Qwen2/2.5/3-VL 使用 rope_3d，输出通常是 [local_output_tokens, out_hidden_size]。输出 token 数等于 t * h * w / spatial_merge_size^2。

5.5 pad、all-gather、去 padding、恢复顺序

max_len_per_rank = max(grouped_pixel_values_len) // embed_dim_reduction_factor

image_embeds_local_padded = pad_to(image_embeds_local, max_len_per_rank)
gathered_embeds = tensor_model_parallel_all_gather(image_embeds_local_padded, dim=0)

rank_embeddings = []
for rank in range(tp_size):
    start_idx = rank * max_len_per_rank
    end_idx = start_idx + grouped_pixel_values_len[rank] // embed_dim_reduction_factor
    rank_embeddings.append(gathered_embeds[start_idx:end_idx])

original_order_embeddings[img_idx] = rank_embed[embed_start:embed_start + img_patches]

tensor_model_parallel_all_gather 要求每个 rank 的 tensor shape 一致，所以先 pad 到所有 rank 中最大的输出长度（vision.py:469-528）。all-gather 后再根据每个 rank 的真实输出 token 数去 padding，并按 image_to_tp_rank 记录的分配顺序拆回原始 item 顺序（vision.py:530-570）。

6. Encoder CUDA Graph + DP

Qwen3-VL 实现了 SupportsEncoderCudaGraph，GPUModelRunner 在 capture 阶段创建 EncoderCudaGraphManager 并捕获预算图（gpu_model_runner.py:6384-6406、6431-6433）。执行时，如果当前 modality 被 manager 支持，会先尝试 CUDA Graph，否则回退到 model.embed_multimodal（gpu_model_runner.py:3044-3057）。

6.1 Qwen3-VL 暴露模型相关元数据

def get_encoder_cudagraph_item_specs(self, mm_kwargs):
    m = self.visual.spatial_merge_size
    grid_thw = self._get_grid_thw_by_modality(mm_kwargs)
    return [
        EncoderItemSpec(
            input_size=t * h * w,
            output_tokens=t * (h // m) * (w // m),
        )
        for t, h, w in grid_thw
    ]

这段把“输入负载”和“输出 embedding 长度”明确告诉 graph manager（qwen3_vl.py:1859-1873）。选择 item 时，Qwen3-VL 也按 grid_thw 计算 cumulative patch offset，从 pixel_values 里切出选中的 item（qwen3_vl.py:1875-1915）。

6.2 捕获图：按 token budget 准备大 buffer

per_mm_item_output = (token_budget + max_batch_size - 1) // max_batch_size
grid_config = [
    [1, spatial_merge_size, per_mm_item_output * spatial_merge_size]
    for _ in range(max_batch_size)
]

dummy_pixel_values = torch.randn(total_patches, flattened_patch_size, ...)
buffers = self.visual.prepare_encoder_metadata(
    grid_config,
    max_batch_size=max_batch_size,
    max_seqlen_override=token_budget * (spatial_merge_size**2),
)

捕获时使用 dummy 输入和 metadata buffer，保证之后 replay 的真实输入能塞进同一组 shape（qwen3_vl.py:1917-1999）。视频场景会构造带帧数的 capture grid，让 cu_seqlens 容量足够。

6.3 Graph manager 内部先 batch pack，再可选 DP shard

EncoderCudaGraphManager 初始化时只有在 mm_encoder_tp_mode == "data" 且 tensor_parallel_size > 1 时启用 use_dp（encoder_cudagraph.py:149-154）。它有两层调度：

函数职责关键行
_execute_local在本 rank 内把多个 item 按 token budget 贪心打包，命中 graph 就 replay，超预算就 eager fallbackencoder_cudagraph.py:337-443
_dp_shard跨 TP rank 按 input_size 做负载均衡，得到本 rank 的 local_mm_kwargsencoder_cudagraph.py:445-507
_dp_gather本地输出 concat 后 pad，TP all-gather，逐 rank 去 padding，再恢复 item 顺序encoder_cudagraph.py:509-574
execute如果 use_dp，依次调用 shard、本地执行、gather；否则只本地执行encoder_cudagraph.py:576-609

所以 CUDA Graph 路径和 eager 路径的 DP 思路完全一致：item 级 shard，本地跑完整 ViT，pad + all-gather，恢复顺序。不同点是本地执行可以走预算图，减少 encoder 调用开销。

7. GPUModelRunner 怎么调 encoder

一次 step 中，GPUModelRunner 先根据 scheduler 输出拿到本步需要计算的多模态输入：

mm_hashes, mm_kwargs, mm_lora_refs = self._batch_mm_inputs_from_scheduler(
    scheduler_output
)
if not mm_kwargs:
    return []

入口是 _execute_mm_encoder（gpu_model_runner.py:2860-2869）。它会把同 modality 的输入尽量 batch 起来，然后执行 encoder：

for modality, num_items, mm_kwargs_batch in group_and_batch_mm_kwargs(...):
    cudagraph_output = None
    if (
        self.encoder_cudagraph_manager is not None
        and self.encoder_cudagraph_manager.supports_modality(modality)
    ):
        cudagraph_output = self.encoder_cudagraph_manager.execute(mm_kwargs_batch)

    if cudagraph_output is not None:
        batch_outputs = cudagraph_output
    else:
        batch_outputs = model.embed_multimodal(**mm_kwargs_batch)

来源：vllm/vllm/v1/worker/gpu_model_runner.py:2984-3057。

无论走 CUDA Graph 还是 eager，最后都会把每个 item 的 encoder 输出写入 encoder_cache：

for mm_hash, output in zip(mm_hashes, encoder_outputs):
    self.encoder_cache[mm_hash] = output

后续 _gather_mm_embeddings 按请求中 placeholder 的 token 范围，从 encoder_cache 中取出本 step 需要的 embedding 片段（gpu_model_runner.py:3094-3134）。接着 runner 统一走 embedding 输入：先对 token id 做文本 embedding，再把多模态位置覆盖为视觉 embedding（gpu_model_runner.py:3385-3400）。底层覆盖逻辑就是 inputs_embeds[is_multimodal] = mm_embeds_flat（vllm/vllm/model_executor/models/utils.py:456-492）。

8. 一个具体 shape 例子

假设 TP=2，一次 encoder batch 有 4 张图，spatial_merge_size=2，patch 数分别为：

patches_per_image = [1000, 100, 200, 50]
output_tokens     = [250, 25, 50, 12]   # floor 视具体 grid 整除情况而定

这个例子展示了两个现实特征：第一，负载按 patch 数平衡，而不是按 item 数平均；第二，通信量按最大 rank 输出长度补齐，所以严重不均衡时 all-gather padding 会浪费带宽。

9. 限制与排查

单 item 不能被进一步拆分。 当前实现按图片/视频 item 分配 rank。如果一个请求只有一张超大图，ViT DP 很难吃满所有 TP rank。
会复制视觉 encoder 权重。 data 模式的收益来自并行处理输入，但每个 TP rank 持有完整视觉权重；小显存卡上要关注峰值显存。
只对支持模型生效。 看模型类是否设置 supports_encoder_tp_data = True，以及是否真的在 _process_*_input 里调用 DP helper。
先看日志和 profiler。 CUDA Graph manager 初始化日志会打印 use_dp；性能上应看到 visual encoder 时间随多 item batch 被 TP rank 分摊。

排查清单

现象优先检查
没有加速tensor_parallel_size 是否大于 1；batch 内是否有多个图片/视频 item；模型是否支持 supports_encoder_tp_data
显存上涨这是预期之一：视觉 encoder 权重在 TP rank 上复制；可降低 TP、限制图片/视频数量或关闭 data
CUDA Graph 没命中检查 cudagraph_mm_encoder、模型是否实现 SupportsEncoderCudaGraph、item token 是否超出预算
多模态 token 数不匹配核对 grid_thw、spatial_merge_size、placeholder 数量和 encoder 输出长度

一句话总结：ViT DP 是“把 TP rank 暂时当作视觉 encoder 的数据并行 worker”。它不改变 LLM 主干的并行方式，也不改变最终 token 序列；它只让视觉输入在进入 LLM 前更快地被编码成可覆盖 placeholder 的 embedding。
