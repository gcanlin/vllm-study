vLLM v1 Prefix Cache 机制详解

本文按 自顶向下 顺序解析 vLLM 的 prefix cache（前缀复用）系统：先看整体架构，再依次深入哈希链 → BlockPool → KVCacheManager → Coordinator（Unitary vs Hybrid） → 各种 SingleType Manager（Full / SlidingWindow / ChunkedLocal / Mamba / CrossAttention）→ Eagle 投机解码与前缀缓存的交互。

0. 整体架构鸟瞰

整个系统是 分层 + 组合 的设计，调用栈从上到下是：

Request  —  用户请求，带 token_ids 调度器拿到
▼
request_block_hasher()  —  把 tokens 按 block_size 切块，做链式哈希 kv_cache_utils.py
▼
KVCacheManager  —  对外 API：get_computed_blocks / allocate_slots / cache_blocks kv_cache_manager.py
▼
KVCacheCoordinator  —  路由：NoPrefix / Unitary / Hybrid kv_cache_coordinator.py
▼
SingleTypeKVCacheManager  —  按 attention 类型实现 find_longest_cache_hit single_type_kv_cache_manager.py
▼
BlockPool  —  物理 block 分配 + hash→block 表 + 双向链表 LRU block_pool.py

1. Block 哈希：缓存的「指纹」

vLLM 把整段 token 序列按 block_size（默认 16）切成多个 block，每个 block 计算一个哈希值，作为 prefix cache 的查找键。

1.1 链式哈希（hash chaining）

关键点：每个 block 的 hash 都依赖它前一个 block 的 hash。这样可以保证「相同 token 子串、但前缀不同」时不会误命中。

📄 vllm/v1/core/kv_cache_utils.py：541-569
def hash_block_tokens(
    hash_function, parent_block_hash, curr_block_token_ids, extra_keys=None,
) -> BlockHash:
    # 链式：把上一个 block 的 hash 作为这次的 "盐"
    if not parent_block_hash:
        parent_block_hash = NONE_HASH  # 第一个 block 用全局 seed
    curr_block_token_ids_tuple = tuple(curr_block_token_ids)
    return BlockHash(
        hash_function((parent_block_hash, curr_block_token_ids_tuple, extra_keys))
    )
为什么要 extra_keys？ 多模态 hash、LoRA 名、cache_salt、prompt embeds 等也参与哈希，避免不同模态/LoRA 错误共享缓存。

1.2 hash + group_id 打包：支持 hybrid 模型

同一个 token block 在「Full attention 层」和「SlidingWindow 层」里的 KV 是不一样的，所以 key 必须区分。vLLM 的做法是把 group_id 用 4 字节 big-endian 拼到 hash 后面：

📄 vllm/v1/core/kv_cache_utils.py：39-75
BlockHash            = NewType("BlockHash", bytes)
BlockHashWithGroupId = NewType("BlockHashWithGroupId", bytes)

def make_block_hash_with_group_id(block_hash, group_id):
    return BlockHashWithGroupId(
        block_hash + group_id.to_bytes(4, "big", signed=False)
    )

2. BlockPool：物理 block 与 hash 表

BlockPool 维护两个核心数据结构：

  • free_block_queue — 双向链表，O(1) 弹出/删除，做 LRU 驱逐用
  • cached_block_hash_to_block — hash → block 的字典，做 prefix 命中查询

2.1 哈希查找表结构示意

cached_block_hash_to_block (BlockHashToBlockMap)
H₀ + group_id=0 (full attn) → KVCacheBlock(block_id=7)
H₀ + group_id=1 (sliding win) → KVCacheBlock(block_id=23)
H₁ + group_id=0 → KVCacheBlock(block_id=8)
H₂ + group_id=0 → {block_id=12: blk, block_id=88: blk} ← 罕见的哈希冲突，存多个

2.2 查询：必须所有 group 都命中才算命中

📄 vllm/v1/core/block_pool.py：184-209
def get_cached_block(self, block_hash, kv_cache_group_ids):
    cached_blocks = []
    for group_id in kv_cache_group_ids:
        key = make_block_hash_with_group_id(block_hash, group_id)
        block = self.cached_block_hash_to_block.get_one_block(key)
        if not block:
            return None  # 任一 group miss → 整体 miss，不做部分命中
        cached_blocks.append(block)
    return cached_blocks

2.3 touch：命中的 block 增加引用计数

📄 vllm/v1/core/block_pool.py：402-417
def touch(self, blocks):
    for block in blocks:
        if block.ref_cnt == 0 and not block.is_null:
            self.free_block_queue.remove(block)  # O(1)，从 LRU 队列摘掉
        block.ref_cnt += 1
关键不变量： 只有 ref_cnt == 0 的 block 才在 free queue 里，才有资格被驱逐。命中的 block 立刻 touch 出来，避免被回收。

def allocate_slots(
    self, request, num_new_tokens,
    num_new_computed_tokens=0,
    new_computed_blocks=None,
    num_lookahead_tokens=0,        # <-- Eagle 投机预算
    num_external_computed_tokens=0,
    delay_cache_blocks=False,
    ...
):
    # 1. 把超出 attention 窗口的 block 释放掉（如 sliding window）
    self.coordinator.remove_skipped_blocks(request.request_id, total_computed_tokens)

    # 2. touch 上 prefix 命中的 block（ref_cnt++）
    self.coordinator.allocate_new_computed_blocks(...)

    # 3. 分配新 block
    new_blocks = self.coordinator.allocate_new_blocks(...)

    # 4. 把刚算满的 block 写入哈希表（caching）
    if self.enable_caching and not delay_cache_blocks:
        num_tokens_to_cache = min(
            total_computed_tokens + num_new_tokens,
            request.num_tokens,  # 截掉未验证的 draft token，不缓存
        )
        self.coordinator.cache_blocks(request, num_tokens_to_cache)

5. Coordinator：Simple（Unitary） vs Hybrid

vLLM 有 3 种 coordinator，由 get_kv_cache_coordinator() 工厂选择：

Coordinator场景策略
KVCacheCoordinatorNoPrefixCache禁用 prefix cache永远返回 0 命中
UnitaryKVCacheCoordinator单 KV group（所有层都是相同 attention 类型）直接 delegate 给唯一的 SingleTypeManager

def find_longest_cache_hit(self, block_hashes, max_cache_hit_length):
    # Fixed-point iteration：每个 attention type 要么接受当前长度，
    # 要么把长度往下压；任何一组压低则重启。单调下降，必定收敛。
    hit_length = max_cache_hit_length
    hit_blocks_by_group = [None] * num_groups
    eagle_verified = set()

    while True:
        curr_hit_length = hit_length
        for idx, (spec, group_ids, manager_cls) in enumerate(self.attention_groups):
            # Full attention 是「向下封闭」的，可以 skip 重复查
            if isinstance(spec, FullAttentionSpec) and cached_blocks is not None:
                curr_hit_length = curr_hit_length // spec.block_size * spec.block_size
                continue

            use_eagle = idx in self.eagle_attn_group_indices and idx not in eagle_verified
            _max_length = curr_hit_length
            if use_eagle:
                # Eagle 允许多搜一块，后面 pop 掉
                _max_length = min(curr_hit_length + spec.block_size, max_cache_hit_length)

            hit_blocks = manager_cls.find_longest_cache_hit(
                block_hashes=_get_block_hashes(spec),
                max_length=_max_length,
                ...
            )
            curr_hit_length = len(hit_blocks[0]) * spec.block_size

        if curr_hit_length >= hit_length:
            break  # 收敛
        hit_length = curr_hit_length
Simple Hybrid 早退： 当 attention_groups 只有 2 个（1 个 full + 1 个其它），第一轮收敛即可，不需要多轮迭代。

6.1 FullAttentionManager — 从左往右线性扫

最简单：从头开始，一个一个查 hash，遇到第一个 miss 就停。Full attention 是向下封闭的——前 N 块命中 ⟹ 前 N-1 块也命中。

📄 vllm/v1/core/single_type_kv_cache_manager.py：481-526
class FullAttentionManager(SingleTypeKVCacheManager):
    @classmethod
    def find_longest_cache_hit(cls, block_hashes, max_length,
                              kv_cache_group_ids, block_pool, ...
                              use_eagle, alignment_tokens, ...):
        computed_blocks = tuple([] for _ in range(len(kv_cache_group_ids)))
        block_size = kv_cache_spec.block_size
        max_num_blocks = max_length // block_size

        for block_hash in itertools.islice(block_hashes, max_num_blocks):
            if cached_block := block_pool.get_cached_block(block_hash, kv_cache_group_ids):
                for computed, cached in zip(computed_blocks, cached_block):
                    computed.append(cached)
            else:
                break  # 第一次 miss 就退出

        # Eagle 模式：把最后一块 pop 掉，强制重算获取 hidden states
        if use_eagle and computed_blocks[0]:
            for computed in computed_blocks:
                computed.pop()
        return computed_blocks

6.2 SlidingWindowManager — 从右往左扫

class SlidingWindowManager(SingleTypeKVCacheManager):
    @classmethod
    def find_longest_cache_hit(cls, block_hashes, max_length, ..., use_eagle, alignment_tokens):
        sliding_window_contiguous_blocks = cdiv(
            kv_cache_spec.sliding_window - 1, kv_cache_spec.block_size
        )
        if use_eagle:
            sliding_window_contiguous_blocks += 1  # 留一块给 Eagle pop

        max_num_blocks = max_length // kv_cache_spec.block_size
        computed_blocks = tuple(
            [block_pool.null_block] * max_num_blocks   # 初始全 null
            for _ in range(len(kv_cache_group_ids))
        )

        num_contiguous = 0
        for i in range(max_num_blocks - 1, -1, -1):  # 右→左
            if cached := block_pool.get_cached_block(block_hashes[i], kv_cache_group_ids):
                for computed, c in zip(computed_blocks, cached):
                    computed[i] = c
                num_contiguous += 1
                if num_contiguous >= sliding_window_contiguous_blocks:
                    break  # 凑够窗口就停
            else:
                num_contiguous = 0   # 断了就重新计数

6.3 ChunkedLocalAttentionManager — Llama 4 风格

把序列切成固定 chunk，attention 只看当前 chunk 内的 token（Llama 4 长上下文优化）。算法类似 SlidingWindow，但按 chunk 边界对齐。注意：这个 manager 不支持 Eagle。

6.4 MambaManager — 状态空间模型

Mamba 不是 attention，存的是整体 hidden state（每个序列 1 个 state block）。所以 prefix cache 行为很特殊：只能复用「完全一样的 prefix」对应的最后一个 state block。

6.5 CrossAttentionManager — encoder-decoder

Whisper 等模型用：encoder KV 在整个 decoding 过程中是固定的，作为 cross-attention 的 key/value。逻辑上跟 Full 类似，但跟着 encoder 输入哈希。

7. Eagle Speculative Decoding 与 Prefix Cache

Eagle 是 vLLM 的投机解码实现：用一个小的 draft 模型预测多个 token，主模型一次性验证。它和 prefix cache 有 两个 关键交互点。

7.1 交互点 ①：lookahead 槽位预留

allocate_slots(num_lookahead_tokens=K) 会多分 K 个 token 的空间，给 draft token 用。这部分不进 cache 表，因为 draft token 未经主模型验证，万一被回滚就成了脏数据。

7.2 交互点 ②：Eagle 的 "pop 最后一块" 技巧

Eagle 的 draft 模型需要主模型最后一层的 hidden states作为输入。但 prefix cache 命中时，整个 block 都没经过 forward，hidden states 不在 GPU 上。

解决方案：找到最长命中后，主动 pop 掉最后一块，让主模型重新算这一块，顺便把 hidden states 留下来给 Eagle 用。

Hybrid 模型里只有部分 attention group 会跑 Eagle（一般是 full attention 那组）。HybridCoordinator 给这些组在 max_length 上多加 1 个 block_size，让它们多扫一块再 pop：

8. 完整请求生命周期回放

① 入队：
scheduler 拿到 Request，调用 request_block_hasher() 算出 block_hashes 列表
② 探测命中：
scheduler 调 kv_cache_manager.get_computed_blocks(request)
③ 分配：
scheduler 调 allocate_slots(...)，touch 命中块 + 分新块 + 预留 lookahead
④ 前向：
主模型 forward 「new」 段；如果开 Eagle，draft 模型 forward 「lookahead」
⑤ 回写缓存：
cache_full_blocks() 把已经算满的 block 插入 hash 表（截掉未验证的 draft）
⑥ 完成/释放：
请求结束 → block ref_cnt-- → 归 0 时挂回 free_block_queue 尾部（LRU）

10. 调试 / 观察的小建议

  • 开 VLLM_LOG_STATS=1，看 prefix_cache_stats 的 hit ratio
  • 跑实验时设 PYTHONHASHSEED=0，保证哈希值跨进程一致（cache_salt 也可控制）
  • Hybrid 模型如果 cache hit 总是 0，先确认是不是哪个 group 的 block_size 没对齐到 LCM
  • Eagle 命中长度永远比预期短 1 块，是设计如此不是 bug
