1. 背景与动机：为什么 Qwen3-Omni 要做流式输入

Qwen3-Omni 的目标是"实时多模态对话"：用户一边说话/说视频，模型一边理解一边生成语音回答。 如果走传统"请求级"接口，浏览器必须先等用户说完一整段，再上传整段音频文件，再等模型生成完毕。 这会引入数秒级延迟，而且 KV 缓存只能从头算一次。

vllm-omni 在 vllm_omni/entrypoints/openai/realtime_connection.py 实现了一个 OpenAI 风格的 /v1/realtime WebSocket 端点，配合 vllm_omni/entrypoints/async_omni.py 的 generate(prompt=AsyncGenerator…) 入口， 让"输入"自身成为异步生成器。Thinker 可以在用户还没说完时就开始 prefill 一段，并把语音 / 视觉 token 逐步推进到 Talker。

这里要区分四种"流"，本文只覆盖前两种：

层次含义本文是否覆盖
protocol-level streamingWebSocket / SSE 协议本身能否多帧是（/v1/realtime 与 /v1/audio/speech/stream）
engine-level streaming input引擎是否允许"同一 request_id 追加更多 prompt 段"是（add_streaming_update_async / resumable=True）
streaming outputtoken / 音频是否边出边返另文 Omni Streaming
async chunkThinker → Talker → Code2Wav 内部的 stage 间分块交付另文 Omni Streaming
关键边界： "engine-level streaming input" ≠ "async_chunk"。前者描述的是浏览器侧能不能不断推新音频； 后者描述的是 stage 之间能不能不等上一 stage 完全结束就把中间产物切片送下去。Qwen3-Omni 实时对话要求两者同时为真。

2.3 文本/视频/图像也走同一通路

Qwen3-Omni 的 multimodal processor 同时声明了 audio / image / video 三种模态都没有上限（get_supported_mm_limits 返回 {"audio": None, "image": None, "video": None}），所以同一个 WebSocket session 中可以混合：

  • 文本：发 {"type":"conversation.item.create", "item":{"content":[{"type":"input_text","text":"…"}]}} 时由上游 vllm 协议层翻译
  • 视频：每个新到的视频片段以 <|vision_start|><|video_pad|><|vision_end|> 占位 token 形式注入 prompt
  • 音频：上文 input_audio_buffer.append，最终在 Thinker 内被 WhisperFeatureExtractor 转为 mel

占位串来自 qwen3_omni_moe_thinker.py:1101-1107 的 get_placeholder_str。

Node 4 · Thinker 端：多模态预处理

每段的 chunk.prompt 进入 Thinker 时由 Qwen3OmniMoeThinkerMultiModalProcessor._call_hf_processor 处理， 其中音频路径会把 PCM 补齐到 hop_length 的整倍数：

qwen3_omni_moe_thinker.py:649-682

class Qwen3OmniMoeThinkerMultiModalProcessor(
    Qwen2_5OmniThinkerMultiModalProcessor,
):
    def _call_hf_processor(self, prompt, mm_data, mm_kwargs, tok_kwargs):
        mm_data = dict(mm_data)
        audios = mm_data.pop("audios", [])

        def pad_to_hop_length(x, hop_length):
            length = x.shape[-1]
            if length % hop_length != 0:
                pad_length = hop_length - (length % hop_length)
                x = np.pad(x, (0, pad_length), mode="constant", constant_values=0)
            return x

        feature_extractor = self.info.get_feature_extractor(**mm_kwargs)
        hop_length = feature_extractor.hop_length      # 160
        if audios:
            mm_data["audio"] = [
                pad_to_hop_length(audio, hop_length)
                if isinstance(audio, np.ndarray)
                else (pad_to_hop_length(audio[0], hop_length), audio[1])
                for audio in audios
            ]

Node 5 · Thinker LLM 前向：每段 prefill + 若干 decode 行

Thinker 是 StageExecutionType.LLM_AR（自回归），所以一段 prompt 进来：

段 1（首段）

prefill 行数 = P₁（音频 + 模板）
decode 行数 = 0..k（取决于何时被段 2 打断）

段 2..4（中间段）

追加 prefill 行数 = ΔPᵢ
continue decode 行数 = 0..kᵢ

段 N（final）

追加 prefill 行数 = ΔP_N
解码到 stop_token，
触发 is_finished=True

每次 Thinker 跑完一步，pooling_output 中包含两层关键 hidden states：

键来源用途
hidden_states.layers[0]word embedding（layer 0）Talker 的"文本 embedding"输入
hidden_states.layers[24]accept_hidden_layer（layer 24）Talker 的条件 hidden state

两个常量在 qwen3_omni.py:36-37 写死：_EMBED_LAYER_KEY="0"、_HIDDEN_LAYER_KEY="24"。

Node 7 · 段边界处理：_construct_thinker2talker_streaming_input_async_chunk

这是 Qwen3-Omni 独有的"streaming input 段缓存"实现。来自上游 _send_single_request 的回调， 通过 pipeline.py 注册的 async_chunk_process_next_stage_input_func 指到这里。核心 3 个分支：

分支 A：thinker_emb.shape[0] > 1

说明这一步 Thinker 算了 多行——典型是新追加段的 prefill。 此时只把 prefill 行缓存到 _pending_streaming_prefills[req_id]，不发送下游：

payload = OmniPayloadStruct(
    meta=MetaStruct(finished=finished),
    embed=EmbeddingsStruct(prefill=emb_cpu),
    hidden_states=HiddenStatesStruct(output=hid_cpu),
    ids=IdsStruct(
        all=request.all_token_ids[-new_prompt_len-1:],
        prompt=request.prompt_token_ids[-new_prompt_len:],
    ),
    speaker=speaker, language=language,
)
transfer_manager._pending_streaming_prefills[req_id] = to_dict(payload)
return None

分支 B：thinker_emb.shape[0] == 1

说明这是一行 decode 输出。如果上一段的 prefill 还缓存着，就把缓存的 prefill 与当前 decode 拼起来一起发：

save_payload = transfer_manager._pending_streaming_prefills.pop(req_id, None)
if save_payload is not None:
    return OmniPayloadStruct(
        meta=MetaStruct(finished=finished),
        embed=EmbeddingsStruct(
            prefill=torch.cat((saved_prefill, emb_cpu), dim=0)),
        hidden_states=HiddenStatesStruct(
            output=torch.cat((saved_output, hid_cpu), dim=0)),
        ids=IdsStruct(
            all=save_payload["ids"]["all"],
            prompt=save_payload["ids"]["prompt"]),
        speaker=speaker, language=language,
    )

分支 C：output_token_ids == []

async_chunk 模式下的"占位 token 步"。不发送，除非 is_finished=True，那时仍要带一个空 meta 段触发下游清理：

if not is_finished:
    # placeholder step, drop
    return None
return OmniPayloadStruct(
    meta=MetaStruct(finished=finished),
    embed=EmbeddingsStruct(decode=emb_cpu),
    hidden_states=HiddenStatesStruct(output=hid_cpu),
    speaker=speaker, language=language,
)
为什么需要缓存？ Talker 期待的 chunk 形状是「[prefill_rows + 1_decode_row, D]」。 而 Thinker 每段的第一步只有 prefill 行（没有 decode），如果直接发出去 Talker 拿到的 ids 和 tensor 行数会少 1，导致后续 codec 解码错位。 缓存到下一次 decode 再拼，保证 「ids 与 tensor 行严格对齐」。

Node 8 · 调度器（OmniGenerationScheduler）的"等待新段"分支

段尾意味着 prompt 暂时被消费完，下一段 input 还没到。调度器避免空转的方式是：

omni_generation_scheduler.py:103-116

num_computed_tokens = request.num_computed_tokens
required_tokens = len(request.prompt_token_ids) - num_computed_tokens
if not self.scheduler_config.enable_chunked_prefill and required_tokens > token_budget:
    break
# async_chunk: don't schedule placeholder tokens when no new chunk is available.
if required_tokens <= 0:
    if self.chunk_transfer_adapter is not None and \
       self.chunk_transfer_adapter.is_done_receiving_chunks(request.request_id):
        self._pending_finish_reqs.append(request)
    req_index += 1
    continue

直觉： required_tokens <= 0 意味着 prompt 里没有新 token 要 prefill，且 streaming 模式下也没新 decode 触发—— 这一轮就跳过它。新段一来 prompt 又增长，required_tokens 重新变正，调度才恢复。 对外效果就是「输入断开时 GPU 不浪费一次 forward」。

5.2 AsyncOmni._add_streaming_input_request

关键三步：

  1. 校验 sampling params（流式输入对参数有限制）：
async_omni.py:484-494

@staticmethod
def _validate_streaming_input_sampling_params(params: OmniSamplingParams) -> None:
    if (
        not isinstance(params, SamplingParams)
        or params.n > 1
        or params.output_kind == RequestOutputKind.FINAL_ONLY
        or params.stop
    ):
        raise ValueError(
            "Input streaming is currently supported only for SamplingParams "
            "with n == 1, output_kind != FINAL_ONLY, and without stop strings."

为什么这三条？

  • n > 1 会让一个 request 有多个序列分支，但流式输入要不断 append 同一个 request 的 prompt，分支语义不清；
  • FINAL_ONLY 只在结束时回包，违背"边出边返"目的；
  • stop strings 要扫描文本，但流式 decode 还没完整文本可扫，需要走 token id stop。
  1. 首段 / 后续段二分支提交：见 Node 3 的代码，关键是 has_submitted_first_chunk 这个 nonlocal 标志。
  2. finally 段保证收尾：

def get_mm_max_tokens_per_item(self, seq_len, mm_counts=None):

                              if m in requested_modalities})

    if "audio" in requested_modalities:
        audio_tokens = Qwen2AudioProcessingInfo.get_mm_max_tokens_per_item(
            self, seq_len=seq_len, mm_counts=mm_counts)
        if audio_tokens is None:
            feature_extractor = self.get_feature_extractor()
            max_audio_samples = feature_extractor.chunk_length * feature_extractor.sampling_rate
            max_audio_tokens = int(
                _get_feat_extract_output_lengths(torch.tensor([max_audio_samples])).item())
            audio_tokens = {"audio": max_audio_tokens}
        mm_max_tokens["audio"] = audio_tokens["audio"]

    return mm_max_tokens

三个细节：

  • {"audio": None, ...} 中的 None 表示每条请求里该模态条目数不限——这就是为什么同一 WebSocket session 可以混 N 段音频；
  • 音频 max tokens 推导：chunk_length × sampling_rate = 一段最长音频样本数；_get_feat_extract_output_lengths 再下采样成 token 数；
  • image/video 完全复用 Qwen2_5_VLProcessingInfo，所以视频流的"按帧抽取 + ViT 切 patch"路径在这里继承下来。

5.4 _construct_thinker2talker_streaming_input_async_chunk 的状态机

def _construct_thinker2talker_streaming_input_async_chunk(
    is_finished: bool, request, thinker_emb, thinker_hid, transfer_manager,
) -> OmniPayloadStruct | None:
    request_id = request.external_req_id
    output_token_ids = _ensure_list(request.output_token_ids)
    speaker = extract_speaker_from_request(request)
    language = extract_language_from_request(request)
    finished = torch.tensor(is_finished, dtype=torch.bool)
    emb_cpu = thinker_emb.detach().cpu()
    hid_cpu = thinker_hid.detach().cpu()

    if output_token_ids:
        if thinker_emb.shape[0] > 1:
            # new streaming input segment is added; cache prefill
            new_prompt_len = thinker_emb.shape[0]
            payload = OmniPayloadStruct(
                meta=MetaStruct(finished=finished),
                embed=EmbeddingsStruct(prefill=emb_cpu),
                hidden_states=HiddenStatesStruct(output=hid_cpu),
                ids=IdsStruct(
                    all=_ensure_list(request.all_token_ids[-new_prompt_len - 1 :]),
                    prompt=_ensure_list(request.prompt_token_ids[-new_prompt_len:]),
                ),
                speaker=speaker, language=language,
            )
            transfer_manager._pending_streaming_prefills[request_id] = to_dict(payload)
            return None
        else:
            # decode step: flush cached prefill (if any) together
            save_payload = transfer_manager._pending_streaming_prefills.pop(request_id, None)
            if save_payload is not None:
                ...  # 见 Node 7 分支 B
            return OmniPayloadStruct(
                meta=MetaStruct(finished=finished),
                embed=EmbeddingsStruct(decode=emb_cpu),
                hidden_states=HiddenStatesStruct(output=hid_cpu),
                ids=IdsStruct(output=output_token_ids),
                speaker=speaker, language=language,
            )
    else:
        if not is_finished:
            return None    # async_chunk placeholder step
        return OmniPayloadStruct(
            meta=MetaStruct(finished=finished),
            embed=EmbeddingsStruct(decode=emb_cpu),
            hidden_states=HiddenStatesStruct(output=hid_cpu),
            speaker=speaker, language=language,
        )

按代码顺序读：

  • 先把 emb / hid 拷到 CPU——后续会跨进程序列化送 Talker，必须在 CPU；
  • 外层 if output_token_ids: 判断是否已有 decode token；这是"流处于 decode 阶段"的近似判据；
  • 分支 A（多行 emb）：这一步 Thinker 同时算了多行 prefill，new_prompt_len 就是新追加的 prompt 长度。把这些行连同 ids.all/prompt 切片缓存——切片用 [-new_prompt_len - 1 :] 多保留 1 个 token 是为了 all 包含最后一个 decode 上下文 token；
  • 分支 B（单行 emb + 有缓存）：torch.cat((saved_prefill, emb_cpu), dim=0) 把上一段的 prefill 与本步 decode 拼起来一并下发；
  • 分支 C（无输出 token）：这是 async_chunk 调度模式下"调度但没产 token"的占位步，直接 return None； 但当 is_finished=True 时仍需带 meta.finished 通知 Talker 收尾。
段缓存的 key 是 external_req_id（不是内部的 request_id）——因为 resumable 实时会话跨段共用同一个外部 id，这样缓存才能跨 segment 找到。

5.5 OmniChunkTransferAdapter 中段的"resumable"语义

Adapter 在收到下游 payload.meta.finished=True 时的处理：

if payload_finished:
    self.finished_requests.add(req_id)
    request.resumable = False
if payload_segment_finished:
    self.segment_finished_requests.add(req_id)

注意 request.resumable = False 是下游 stage 在确认上游真的彻底结束后才能改的。这保证了：

  1. 段中间收到 finished=False, segment_finished=True ⇒ 只把 req_id 加入 segment_finished_requests，调度器把它当作"暂停"而不是销毁；
  2. final=true 段处理完 ⇒ finished=True ⇒ 同时清掉 resumable，调度器才会进入 _pending_finish_reqs 真正回收。

6. 总结：流式输入的边界与不变量

  1. request_id 在整个 realtime session 保持不变。这是 KV cache 与多段 prompt 能拼接的前提；浏览器侧不可重新分配。
  2. "段（segment）"与"请求（request）"是两层语义。
    • 段：一次 input_audio_buffer.append 或一次文本/视频追加 ⇒ 一个 streaming_update；
    • 请求：直到 final=true 才结束，resumable 在中间一直为 True。
  3. 多模态在协议层透明。文本 / 音频 / 视频在 chunk.prompt 里以 placeholder token 形式表达，Thinker 用同一套 MultiModalProcessor 处理；区别只在 ViT/Audio encoder 不同。
  4. 音频长度必须是 hop_length (=160) 的整倍数。Processor 自己会 pad，但客户端如果做 VAD 也最好按帧切。
  5. async_chunk 路径有 3 类 step：多行 emb（缓存）、单行 emb（冲洗）、空 emb（占位）；只有冲洗才真正下发到 Talker。
  6. 调度器主动跳过 "required_tokens ≤ 0" 的请求。否则段间会出现"空 forward"，浪费一次 GPU step。
  7. 不要把 is_segment_finished 当 is_finished 用。前者是"输入段结束"，后者是"整个 request 彻底结束"——把段当请求清理会丢掉跨段的缓存与映射。
  8. Streaming input ≠ Streaming output。本文只讲入端；输出端的 response.audio.delta / Talker→Code2Wav 的 async chunk 行为见 vLLM-Omni Streaming I/O + async_chunk。

如果你要新增一个 Omni 模型且想复用 streaming input：（1）在 vllm_omni/model_executor/stage_input_processors/ 新增一个 thinker2*_async_chunk 函数，处理"prefill 多行缓存 + decode 行冲洗"的同样模式； （2）在 pipeline 的 StagePipelineConfig 里同时填 custom_process_next_stage_input_func 与 async_chunk_process_next_stage_input_func；（3）保证 _construct_*_streaming_input_async_chunk 的 key 使用 request.external_req_id 以跨段对齐。
