1. PR #3614 先看结论

这个 PR 的目标不是普通“输出 streaming”，而是让 /v1/realtime 的音频输入在 async_chunk=true 时可以早启动：客户端不等整段音频上传完，Qwen3-Omni 把音频按约 5 秒切片，每片作为同一个 logical session 的 sub-request 进入 Stage-0 Thinker。每片 Thinker 结束后可以推进 Talker/Code2Wav 输出音频，下一片继续复用上一片的会话记忆。

input slice 0 -> output segment 0 -> input slice 1 -> output segment 1 -> ... -> final slice -> final output

2. vLLM-Omni 里有哪些流式

名字它流的是什么是不是改变模型语义核心代码
OpenAI Realtime websocket 输入客户端发 input_audio_buffer.append，server 把 PCM16 chunk 放进 audio_queue。只是传输层 buffer；是否早启动取决于后面的 engine streaming input。上游 vllm/entrypoints/speech_to_text/realtime/connection.py；Omni 覆写输出处理在 vllm_omni/entrypoints/openai/realtime_connection.py
Engine streaming inputAsyncGenerator[StreamingInput] 逐个 yield TokensPrompt 音频片。是。每个片段是同一 session 的一次 continuation，推理可以夹在输入片之间。AsyncOmni.generate，_add_streaming_input_request，AsyncOmniEngine.add_streaming_update
Scheduler streaming session同一 Request 被 StreamingUpdate 更新，状态从 WAITING_FOR_STREAMING_REQ 回到 WAITING。是。Stage-0 AR 会把旧输出并入新 prompt；下游 generation 通常替换 prompt。OmniARScheduler._update_request_as_session，OmniGenerationScheduler._update_request_as_session
用户可见流式输出text delta、audio delta、done 事件。不改变模型语义，只改变返回粒度。RealtimeConnection._run_generation，_raw_waveform_to_deltas，serving_chat.py 的 SSE path
async_chunkstage-to-stage 的 connector chunk：Thinker hidden/embed，Talker codec codes，Code2Wav 音频。理论上不该改变完整 request 语义；但和 streaming input 组合后，每个输入片本身就是新语义边界。OmniChunkTransferAdapter，OmniConnectorModelRunnerMixin.send_chunk/recv_chunk，Qwen3 stage processors
codec/audio chunkingTalker 产生 codec frame 后按 codec_chunk_frames 和 left context 切给 Code2Wav。通常只影响下游 vocoder 的增量解码与 TTFA。talker2code2wav_async_chunk，Code2Wav streaming decode
chunked prefill完整输入已知，只是 scheduler 把 prefill tokens 分批 schedule。不应改变 request 语义。vLLM scheduler enable_chunked_prefill，不是 #3614 的 streaming input

3. 整体框架图

4. 流式输入时序

4.1 websocket 到 engine

Client
  | input_audio_buffer.append(audio_0)
  v
RealtimeConnection.audio_queue.put(audio_0)
  | input_audio_buffer.commit(final=false)
  v
start_generation()
  | creates audio_stream_generator()
  | creates input_stream queue for stage-0 generated token feedback
  v
OpenAIServingRealtime.transcribe_realtime()
  | model_cls.buffer_realtime_audio(audio_stream, input_stream, model_config)
  | renderer.render_cmpl_async(...)
  v
AsyncOmni.generate(prompt=AsyncGenerator[StreamingInput])

Qwen3-Omni 的 buffer_realtime_audio 用 Qwen3ASRRealtimeBuffer 缓冲音频。PR 版本里，async_chunk=true 时每凑够约 5 秒就 yield 一个 TokensPrompt(prompt_token_ids=..., multi_modal_data={"audio": segment})；async_chunk=false 时为避免 full-payload 路径重复响应，中间片不会 yield，只在 flush() 时发剩余音频。

4.2 engine 到 scheduler

AsyncOmni._add_streaming_input_request()
  first chunk:
    AsyncOmniEngine.add_request_async(..., resumable=True)
      -> StageSubmissionMessage(type="add_request")
      -> Orchestrator._handle_add_request()
  later chunks:
    AsyncOmniEngine.add_streaming_update_async(..., resumable=True)
      -> StageSubmissionMessage(type="streaming_update")
      -> Orchestrator._handle_streaming_update()
  final sentinel:
    TokensPrompt([0]) with resumable=False
      -> marks whole streaming session terminal
_validate_streaming_input_sampling_params() 当前限制：SamplingParams、n == 1、output_kind != FINAL_ONLY、不允许 stop strings。这个限制是为了让 session continuation 和 delta 输出状态机更简单。

5. async_chunk 时序

5.1 不开 async_chunk：Orchestrator 显式转发

Stage-0 Thinker segment done
  -> EngineCoreOutput(is_segment_finished=True)
  -> Orchestrator._route_output()
       if streaming.enabled and segment_finished:
         next_client.process_engine_inputs(..., streaming_context=req_state.streaming)
         next_pool.submit_initial/update(...)
  -> downstream stage sees normal request/update

这条路径的关键点是 streaming_context.new_prompt_len_snapshot。Stage-0 可能已经把下一片 prompt 追加进 request；thinker2talker() / thinker2talker_token_only() 会用这个 snapshot 去掉“下一片新 prompt tail”，避免把下一片输入错算进上一片 Talker 输入。

5.2 开 async_chunk：下游先 prewarm，数据靠 connector chunk

first streaming slice
  Orchestrator._handle_add_request()
    -> submit Stage-0 real request
    -> _prewarm_async_chunk_stages()
         Stage-1/2 submit placeholder prompt
         downstream request.resumable=True

Stage-0 executes
  OmniARScheduler.update_from_output()
    -> chunk_transfer_adapter.save_async(pooler_output, request, is_segment_finished)
    -> custom processor: thinker2talker_async_chunk()
    -> connector.put(external_req_id_stage_chunk_id, payload)

Stage-1 scheduler loop
  chunk_transfer_adapter.process_pending_chunks()
    -> request WAITING_FOR_CHUNK
    -> connector.get(...)
    -> request.additional_information = payload
    -> request.status = WAITING / RUNNING

current slice ends
  payload.meta.is_segment_finished = True
  downstream stops polling this segment

next input slice arrives
  Orchestrator._handle_streaming_update()
    -> Stage-0 streaming_update
    -> _prewarm_async_chunk_stages() again
  downstream scheduler clears segment_finished_requests
  next connector chunk resumes the same logical session
字段含义为什么需要
request.resumable这是一个 streaming session，当前 segment 结束后还可能继续。Request finish 后不要立刻整体释放；下游 prewarm request 也要知道后续能继续。
meta.is_segment_finished当前音频片结束。让下游停止等待本片更多 chunk，但不要清掉整个 request。
meta.finished整个 streaming request 结束。只有它为真，才走 terminal cleanup。
num_output_placeholdersvLLM async scheduling 预占但尚未确认的输出 token。PR 用 confirmed token cursor 发送 chunk，避免把未确认 token/embedding 传下游。

6. 代码走读主线

6.1 RealtimeConnection：输入启动和输出 delta

vllm_omni/entrypoints/openai/realtime_connection.py 继承上游 realtime connection。输入侧还是上游逻辑：append 解码 PCM16 入队，commit(final=false) 启动 generation，commit(final=true) 放入 None sentinel。Omni 覆写的重点在 _run_generation()：

  • 调用 self.engine.generate(prompt=streaming_input_gen, request_id=..., sampling_params_list=...)。
  • 只把 stage_id == 0 的 text delta 回传为 TranscriptionDelta，并把 Stage-0 新 token id 放入 input_stream，供下一轮 realtime prompt 构造上下文。
  • _extract_audio_chunks() 支持 multimodal_output["audio"] 或 ["model_outputs"]；_raw_waveform_to_deltas() 同时兼容“累计 waveform”和“真 delta waveform”。
  • 最后总是发 response.audio.done，避免客户端挂住。

6.2 AsyncOmni / AsyncOmniEngine：把 generator 变成 request update

AsyncOmni.generate() 检测 prompt 是 AsyncGenerator 后进入 _add_streaming_input_request()。第一片音频用 add_request_async(..., resumable=True)，后续片用 add_streaming_update_async(..., resumable=True)。generator 结束时，如果已经提交过首片，就发一个 TokensPrompt([0]) 且 resumable=False 的 final update。

AsyncOmniEngine._build_add_request_message() 负责把 prompt 经过 stage-0 input processor 变成 OmniEngineCoreRequest，并把 resumable 写进 request；StageSubmissionMessage.type 只有两个值：add_request 和 streaming_update。

6.3 Orchestrator：streaming state 与 stage 路由

  • OrchestratorRequestState.streaming 保存 enabled、segment_finished、new_prompt_len_snapshot、bridge_states。
  • _handle_add_request() 用 prompt.resumable 打开 streaming flag；PR #3614 让 async_chunk 模式在首片后调用 _prewarm_async_chunk_stages()。
  • _handle_streaming_update() 找到现有 request state 后调用 StagePool.submit_update()；PR #3614 新增：streaming update 时也会 prewarm downstream stages，并把 downstream request 标为 resumable。
  • _orchestration_loop() 从 raw EngineCoreOutput 把 is_segment_finished 和 new_prompt_len_snapshot 写回 request state。
  • _route_output() 在 non-async_chunk 路径下，遇到 finished 或 streaming segment finished 就 _forward_to_next_stage()；如果是 streaming session 且整条 request finished，会额外发一个 resumable=False terminal update 给下游。

6.4 Scheduler：session update 怎么改 Request

OmniARScheduler._update_request_as_session() 是 Stage-0 Thinker 的核心。它先记录新片 prompt 长度到 _new_prompt_len_snapshot，再处理 async scheduling 的未确认 placeholder：如果 num_output_placeholders > 0，设置 discard_latest_async_tokens=True，回退 num_computed_tokens，清空 spec tokens，避免把上一片末尾尚未确认的 token 带入下一片。

Stage-0 后续调用 upstream session update 逻辑，把“旧 prompt + 已确认输出 + 新片 prompt”拼成新的 prompt。下游 async_chunk stage 不在 update 时替换 prompt，因为真实 prompt/additional information 要等 connector chunk 到来；它只清掉 segment_finished_requests 并把状态放回 WAITING。

OmniGenerationScheduler._update_request_as_session() 是 Code2Wav/generation 类 stage 的替换语义：清空 _output_token_ids 和 _all_token_ids，把 update prompt 作为新 prompt，num_computed_tokens=0，并刷新 block hashes。

6.5 Connector / async_chunk：chunk 生产、接收、finish marker

OmniChunkTransferAdapter.save_async() 计算 confirmed watermark：num_computed_tokens - num_output_placeholders。这就是 PR 里“不能把 async scheduling 多产出的最后一个 placeholder embedding 传给下游”的落点。_send_single_request() 会把 custom processor 产出的 OmniPayloadStruct 送到 connector，同时强制写入两个 meta：

payload_data.meta.finished = is_finished
payload_data.meta.is_segment_finished = is_segment_finished

_poll_single_request() 接收 chunk 后，如果是 AR 下游，直接把 payload 放进 request.additional_information；如果 chunk_id > 0 and request.resumable，会调用 construct_next_stage_streaming_input_prompt()。这个 helper 不重置 num_computed_tokens，而是把已计算输出移动进 prompt_token_ids，再追加按 ids.prompt 算出的新 placeholder prompt，并刷新 block hashes。

6.6 Qwen3-Omni stage processors：Thinker -> Talker -> Code2Wav

thinker2talker_async_chunk() 在 chunk 0 处理首个 prefill，后续如果 request.resumable 走 _construct_thinker2talker_streaming_input_async_chunk()：

  • 当 thinker_emb.shape[0] > 1，说明新 streaming input segment 带来了 prefill embeddings/hidden states。它先缓存到 _pending_streaming_prefills[request_id]，返回 None，不立刻发下游。
  • 下一次 decode chunk 到来时，把缓存的 prefill 与当前 decode embedding 拼起来发送，保证 Talker 的 ids 和 tensor rows 对齐。
  • 没有 output token 且未 finish 时返回 None，避免把 async placeholder 或 stop token 的 embedding/hidden state 发下游。

_get_streaming_talker_tokens() 用 last_prompt_len / last_output_len 只取本 segment 增量；_get_streaming_codec_delta_len() 用 talker2code2wav_last_seq_len 只取新 codec 帧。talker2code2wav_async_chunk() 再按 codec_chunk_frames 聚合 codec rows，并携带 left_context_size 供 Code2Wav 做流式解码。

6.7 Model runner：流式 prefill 的 additional info

OmniGPUModelRunner 遇到同一个 req_id 再次出现在 scheduled_new_reqs 时，走 _update_streaming_request()；PR #3614 把 _update_streaming_input_additional_info() 简化成只设置 meta.num_processed_tokens = 0 和 meta.resumable = True。Qwen3 Talker 的 talker_preprocess() 看到 meta.resumable 时，不再用 runner 的 num_computed_tokens 覆盖 meta.num_processed_tokens，避免 streaming prefill 被错误当成已处理过。

7. 关键状态与不变量

状态位置不变量
external_req_idOmniRequest / connector keyconnector put/get key 用 external id 跨 stage 对齐；内部 request id 可带随机 suffix。
resumableEngineCoreRequest / Requestsegment finish 后不代表 request terminal；下游 prewarm request 也必须保留这个 flag。
_new_prompt_len_snapshotOmniARScheduler -> EngineCoreOutput -> Orchestrator streaming statenon-async_chunk 转发时，用来从 Thinker prompt 里切掉刚追加的下一片输入。
num_output_placeholdersvLLM async scheduling request state发送 chunk 或拼接下一片时只能使用 confirmed tokens。
segment_finished_requestsOmniChunkTransferAdapter只暂停当前 segment 的 chunk polling；下一次 streaming update 要 discard。
_pending_streaming_prefillsQwen3 Thinker->Talker async processor新 segment prefill chunk 先缓存，等下一次 decode chunk 一起发送，保持 tensor/id 对齐。
requests_num_chunks_sentOmniChunkTransferAdapter按 confirmed watermark 防重复发送，segment finish 后清理。
meta.finished vs meta.is_segment_finishedMetaStruct / connector payload前者释放整条 request，后者只结束当前音频片。

建议读码顺序

  1. vllm_omni/entrypoints/openai/realtime_connection.py：先看输入启动和输出事件。
  2. vllm_omni/model_executor/models/qwen3_omni/qwen3_omni.py::buffer_realtime_audio：看音频如何切成 prompt。
  3. vllm_omni/entrypoints/async_omni.py::_add_streaming_input_request：看 add_request / streaming_update 分界。
  4. vllm_omni/engine/orchestrator.py：看 request state、prewarm 和 route。
  5. vllm_omni/core/sched/omni_ar_scheduler.py 与 omni_generation_scheduler.py：看 Request 更新规则。
  6. vllm_omni/distributed/omni_connectors/transfer_adapter/chunk_transfer_adapter.py：看 chunk 收发和 finish marker。
  7. vllm_omni/model_executor/stage_input_processors/qwen3_omni.py：最后看 Qwen3 的 tensor/id 对齐细节。
一句话总括：用户输入 streaming 是 logical request 的 continuation；用户输出 streaming 是协议返回粒度；async_chunk 是 stage 间数据面的 streaming；chunked prefill 是 scheduler 粒度优化。PR #3614 的难点就是把这三种“流式”同时放在同一个 request 状态机里，并避免 segment finish 被当成 request finish。
