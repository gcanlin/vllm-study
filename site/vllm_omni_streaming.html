4.2 Node 0 · Client 收到第一个音频 delta 的最短路径

从浏览器侧观察，"按下回车 → 听到第一个字"的链路有 6 跳：

  1. 浏览器发 input_audio_buffer.commit final=false + 第一段 append；
  2. RealtimeConnection 把段缓冲转成 StreamingInput 投入 streaming_input_gen；
  3. AsyncOmni._add_streaming_input_request 用首段调用 add_request_async(resumable=True)；
  4. Orchestrator 把 prompt 推给 Stage 0；Thinker 跑 prefill；
  5. OmniChunkTransferAdapter.save_async 入队 → save_loop 调用 thinker2talker_async_chunk → connector.put；
  6. Stage 1 的 Adapter 在 recv_loop 中 _poll_single_request 取到这一段，唤醒调度器；Stage 1 → Stage 2 重复。

4.3 Node 1 · Adapter 双线程的存在意义

base.py:36-43

self.stop_event = threading.Event()

self.recv_thread = threading.Thread(target=self.recv_loop, daemon=True)

为什么要单独的线程？ 因为 connector.put / connector.get 走的是共享内存（SharedMemoryConnector），底层是 shm_open 系统调用； 如果在 scheduler 主循环里直接调用会阻塞调度。把它们丢到背景线程，主循环只用 load_async / save_async 入队，立刻返回。

4.4 Node 2 · 调度器的 WAITING_FOR_CHUNK 状态

chunk_transfer_adapter.py:494-522

def _process_chunk_queue(self, queue, waiting_for_chunk_list,
                         target_status, finished_load_reqs):
    queue_snapshot = list(queue)
    for request in queue_snapshot:
        if request.status != RequestStatus.WAITING_FOR_CHUNK:
            if request.request_id in self.requests_with_ready_chunks:
                continue
            if self.is_done_receiving_chunks(request.request_id):
                request.additional_information = None
                continue
            self.load_async(request)
            request.status = RequestStatus.WAITING_FOR_CHUNK
        else:
            if request.request_id in finished_load_reqs:
                request.status = target_status
                finished_load_reqs.remove(request.request_id)
                self.requests_with_ready_chunks.add(request.request_id)
                continue
        queue.remove(request)
        self.requests_origin_status[request.request_id] = target_status
        waiting_for_chunk_list.append(request)

每个 scheduler step：

  1. 没在等 chunk 的请求 ⇒ 调 load_async，状态打成 WAITING_FOR_CHUNK，从 running/waiting 移到 waiting_for_chunk_list；
  2. 已经在等 chunk ⇒ 看 _finished_load_reqs（recv_loop 把它写满）；如果命中，恢复到原状态并加入 ready 集合；
  3. scheduler 下一个 step 从 ready 集合中正常调度，prefill 新到的 chunk。

4.5 Node 3 · Stage 2 输出 PCM → WebSocket delta 的去重

RealtimeConnection 在每条 Orchestrator output 上调用 _extract_audio_chunks：

def _extract_audio_chunks(self, output) -> tuple[list[np.ndarray], int]:
    mm = getattr(output, "multimodal_output", None)
    if not isinstance(mm, dict):
        return [], 24000
    sr = mm.get("sr") or mm.get("sample_rate") or mm.get("audio_sample_rate") or 24000
    key = "audio" if "audio" in mm else ("model_outputs" if "model_outputs" in mm else None)
    if key is None:
        return [], int(sr)
    raw_audio = mm.get(key)
    chunks: list[np.ndarray] = []
    if isinstance(raw_audio, (list, tuple)):
        if len(raw_audio) > 0:
            arr = self._tensor_to_numpy(raw_audio[-1])
            if arr is not None and arr.size > 0:
                chunks.extend(self._raw_waveform_to_deltas(arr))
    else:
        arr = self._tensor_to_numpy(raw_audio)
        if arr is not None and arr.size > 0:
            chunks.extend(self._raw_waveform_to_deltas(arr))
    return chunks, int(sr)

然后 _raw_waveform_to_deltas 处理两种 stage 行为：

realtime_connection.py:63-81

def _raw_waveform_to_deltas(self, arr: np.ndarray) -> list[np.ndarray]:
    if arr.size == 0:
        return []
    ref = self._realtime_audio_ref
    if ref is None:
        self._realtime_audio_ref = arr.copy()
        return [arr]
    if self._numpy_audio_prefix_match(ref, arr):
        delta = arr[ref.shape[0] :]
        self._realtime_audio_ref = arr.copy()
        return [delta] if delta.size > 0 else []
    # True per-step delta (not a prefix extension of what we have seen).
    self._realtime_audio_ref = np.concatenate([ref, arr])
    return [arr]

为什么要分两种模式？ 不同 stage runner 在 emit 音频时有两种约定：

模式arr 含义检测方式
累积模式每次 emit 一个不断增长的完整 waveform_numpy_audio_prefix_match：新数组前 N 个 sample 等于上次的全部 ref
delta 模式每次 emit 只包含本步新生成的 PCM前缀不匹配 ⇒ 直接当成 delta

两种模式都被支持，避免客户端因引擎升级或 stage 实现改变而播出"重复音频"。

5. 模块源码逐行解析

5.1 stage_config._select_processor_funcs：怎么选 async vs sync

stage_config.py:792-803

def _select_processor_funcs(
    ps: StagePipelineConfig,
    async_chunk: bool,
) -> tuple[str | None, str | None]:
    """Pick ``(input_proc, next_stage_proc)`` based on the async_chunk mode."""
    next_stage_proc = ps.custom_process_next_stage_input_func
    input_proc = ps.custom_process_input_func
    if async_chunk and ps.async_chunk_process_next_stage_input_func:
        next_stage_proc = ps.async_chunk_process_next_stage_input_func
    elif not async_chunk and ps.sync_process_input_func:
        input_proc = ps.sync_process_input_func
    return input_proc, next_stage_proc

按代码顺序读：

  • "next_stage_proc" 是 「我这个 stage 算完后，怎么把输出变成下一 stage 的 prompt」； "input_proc" 是 「我这个 stage 作为下游，怎么读上游的输出」；
  • 默认走 custom_*（同步完整 payload）；
  • 只有当 async_chunk 开启且该 stage 注册了 async 版本 才覆盖 next_stage_proc；
  • 反之只有当 async_chunk 关闭且该 stage 注册了 sync 版本 才覆盖 input_proc。

这个二维表的好处是：一个 pipeline 配置文件同时声明两套处理函数，部署时由 deploy.async_chunk 切换，不需要改 pipeline 拓扑。Qwen3-Omni 的 pipeline.py 就是典型例子。

5.2 OmniTransferAdapterBase 的 recv_loop / save_loop

base.py:49-83

def recv_loop(self):
    while not self.stop_event.is_set():
        n = len(self._pending_load_reqs)
        any_success = False
        for _ in range(n):
            if not self._pending_load_reqs:
                break
            request = self._pending_load_reqs.popleft()
            request_id = request.request_id
            if request_id in self._cancelled_load_reqs:
                self._cancelled_load_reqs.discard(request_id)
                continue
            self.request_ids_mapping[request_id] = request.external_req_id
            try:
                is_success = self._poll_single_request(request)
                if is_success:
                    any_success = True
                else:
                    self._pending_load_reqs.append(request)
            except Exception as e:
                self._pending_load_reqs.append(request)
                logger.warning(...)

        with self._recv_cond:
            if not self._pending_load_reqs and not self.stop_event.is_set():
                self._recv_cond.wait(timeout=0.1)
            elif not any_success and not self.stop_event.is_set():
                self._recv_cond.wait(timeout=0.001)

按代码顺序读：

  • 每个 pass 只处理"开始时已经在队列里"的请求（n = len(...)），新加入的留到下一 pass，避免饿死取消请求；
  • 从队头弹出 ⇒ 调 _poll_single_request ⇒ 拿到就标 _finished_load_reqs，没拿到就重新挂回队尾；
  • "1 ms back-off" 是关键：纯轮询会把 CPU 烧爆（shm_open 失败也很贵），失败一轮后阻塞 1 ms，给生产方时间；
  • "100 ms back-off"：连请求都没有 ⇒ 长睡，等下次 load_async 调 notify() 唤醒。

save_loop 更简单：

base.py:85-97

def save_loop(self):
    while not self.stop_event.is_set():
        while self._pending_save_reqs:
            task = self._pending_save_reqs.popleft()
            try:
                self._send_single_request(task)
            except Exception as e:
                logger.warning(...)
        with self._save_cond:
            if not self._pending_save_reqs and not self.stop_event.is_set():
                self._save_cond.wait(timeout=0.1)

注意 send 没有失败重试——失败只记日志。原因是 connector.put 在共享内存里要么放进去要么真的就坏了，重试也救不回来。

5.5 RealtimeConnection._run_generation：输出端的总开关

realtime_connection.py:112-175

async def _run_generation(
    self,
    streaming_input_gen: AsyncGenerator,
    input_stream: asyncio.Queue[list[int]],
):
    request_id = f"rt-{self.connection_id}-{uuid4()}"
    sent_audio = False
    audio_done_sent = False
    full_text = ""
    prompt_token_ids_len = 0
    completion_tokens_len = 0
    self._realtime_audio_ref = None

    sampling_params_list = list(self.engine.default_sampling_params_list)
    sampling_params_list = coerce_param_message_types(
        sampling_params_list,
        is_streaming=True,
    )
    try:
        result_gen = self.engine.generate(
            prompt=streaming_input_gen,
            request_id=request_id,
            sampling_params_list=sampling_params_list,
        )

        async for output in result_gen:
            stage_id = getattr(output, "stage_id", None)
            if stage_id == 0 and output.outputs:
                first_output = output.outputs[0]
                new_token_ids = list(first_output.token_ids)
                if new_token_ids:
                    input_stream.put_nowait(new_token_ids)
                if output.prompt_token_ids:
                    prompt_token_ids_len = max(
                        prompt_token_ids_len,
                        len(output.prompt_token_ids),
                    )
                delta_text = first_output.text or ""
                full_text += delta_text
                completion_tokens_len += len(new_token_ids)
                if delta_text:
                    await self.send(TranscriptionDelta(delta=delta_text))

            audio_chunks, sample_rate = self._extract_audio_chunks(output)
            for chunk in audio_chunks:
                sent_audio = True
                await self.send_json({
                    "type": "response.audio.delta",
                    "audio": self._pcm16_b64(chunk),
                    "format": "pcm16",
                    "sample_rate_hz": sample_rate,
                })

逐句看：

  • coerce_param_message_types(..., is_streaming=True) 把 sampling_params 的 output_kind 强制成 DELTA——确保每个 output 只携带新增 token；
  • async for output in result_gen 是整个流式输出循环。每次拿到一个 Orchestrator output；
  • stage_id == 0 走文本路径：把 token 写回 input_stream（给协议层 turn-taking 用），并把 delta_text 包成 TranscriptionDelta 通过 WS 发出；
  • _extract_audio_chunks(output) 处理任意 stage 的 multimodal_output，把 PCM 切成 deltas 发给客户端。

  3. TranscriptionDone 必须带 UsageInfo，因为 OpenAI Realtime 协议要求计费端能拿到 token 计数。

5.6 OmniStreamingSpeechHandler：句级 TTS 流式输出

这是另一个 WS 路径——客户端只发"文本"，服务端每个完整句子合成一段 WAV，并按 audio.start + binary frame + audio.done 发回：

serving_speech_stream.py:122-146

if msg_type == "input.text":
    text = msg.get("text", "")
    if not isinstance(text, str):
        await self._send_error(websocket, "input.text requires a string value")
        continue
    sentences = splitter.add_text(text)
    for sentence in sentences:
        await self._generate_and_send(websocket, config, sentence, sentence_index)
        sentence_index += 1

elif msg_type == "input.done":

    if remaining:
        await self._generate_and_send(websocket, config, remaining, sentence_index)
        sentence_index += 1
    await websocket.send_json({
        "type": "session.done",
        "total_sentences": sentence_index,
    })
    return

按代码顺序读：

  • 客户端可任意分段送 text；SentenceSplitter.add_text 会按句界（句号、问号、感叹号等）切完整句；
  • 切到一句就立刻生成一句，不等剩余文本——这就把"网络延迟 + 生成延迟"重叠了；
  • input.done 时 flush buffer（句末没标点的尾巴），再发 session.done 收尾。

生成内部：

try:

注意 config.stream_audio 决定是同一句话内是否再次细分流出：

  • stream_audio=True ⇒ _generate_pcm_chunks 异步迭代 PCM 帧，每帧立刻 send_bytes；
  • stream_audio=False ⇒ 等到这句全合成完，整段 WAV 一次性发。

6. 总结：边界、不变量与常见坑

  1. 四种"流式"是正交的。开了 async_chunk 不等于自动有流式输入；改 output_kind 才决定流式输出粒度。
  2. async_chunk 是"stage 间"的概念，不影响"stage 内"是否流式 forward。Stage 内还是按 vLLM 调度器的 chunked_prefill / decode 步走。
  3. Adapter 的所有 IO 都在背景线程。Scheduler 主循环只做轻量入队 / 状态推动，避免 shm_open 阻塞调度。
  4. resumable 与 finished 是两个独立 flag。前者是请求级"我会不会再回来"，后者是 stage payload 级"上游真的不再产了"。错把段当请求清理会导致跨段缓存被破坏。
  5. request_id 与 external_req_id 不同。接收侧用 internal id 跟踪调度状态，发送侧用 external id 与下游对齐 connector key；中间通过 request_ids_mapping 互查。
  6. "required_tokens ≤ 0 ⇒ skip" 是 async_chunk 调度器避免空转的关键。要保留这个分支，否则段间 GPU 会被一次次唤起空步。
  7. 音频去重靠 _realtime_audio_ref。支持两种 stage 输出约定（累积 vs 增量）；如果你新接入 stage 输出不一致，会被自动归并为 deltas。
  8. TTS 流式输出粒度有两层：句级（SentenceSplitter）+ 句内（stream_audio）。两者都开才有最小延迟。
  9. 新增 Omni 模型如何接入流式：在 pipeline 注册 async_chunk_process_next_stage_input_func + custom_process_next_stage_input_func 两套； 在每个 stage 的 thinker2*_async_chunk 函数里用 chunk_id == 0 缓存 prefill，后续用 thinker_emb.shape[0] 区分 prefill / decode（流式输入还要用 _pending_streaming_prefills）。
常见坑：
  • 关掉 async_chunk 但忘了写 sync_process_input_func ⇒ 启动期 _validate_pipeline_has_processors 会抛错；
  • 客户端发的 PCM 不是 16 kHz mono ⇒ openai_realtime_client.py 直接 raise；服务端会等不到 audio token 而超时；
  • Stage 输出 cumulative waveform 但忘了清 _realtime_audio_ref ⇒ 下一个 connection 会被前一会话尾巴干扰（_run_generation 入口处的 self._realtime_audio_ref = None 就是为这个）；
  • Stage 上把 is_segment_finished 当成 finished 用 ⇒ adapter 把 sender state 清掉，下一段 prefill 缓存找不到，thinker_emb 与 ids 行数不齐对崩。
