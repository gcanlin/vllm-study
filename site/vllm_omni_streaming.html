1. PR #3614 先看结论

这个 PR 的目标不是普通“输出 streaming”，而是让 /v1/realtime 的音频输入在 async_chunk=true 时可以早启动：客户端不等整段音频上传完，Qwen3-Omni 把音频按约 5 秒切片，每片作为同一个 logical session 的 sub-request 进入 Stage-0 Thinker。每片 Thinker 结束后可以推进 Talker/Code2Wav 输出音频，下一片继续复用上一片的会话记忆。

input slice 0 -> output segment 0 -> input slice 1 -> output segment 1 -> ... -> final slice -> final output

4. 流式输入时序

4.1 websocket 到 engine

  v
RealtimeConnection.audio_queue.put(audio_0)
  | input_audio_buffer.commit(final=false)
  v

  | creates audio_stream_generator()
  | creates input_stream queue for stage-0 generated token feedback
  v
OpenAIServingRealtime.transcribe_realtime()
  | model_cls.buffer_realtime_audio(audio_stream, input_stream, model_config)
  | renderer.render_cmpl_async(...)
  v

Qwen3-Omni 的 buffer_realtime_audio 用 Qwen3ASRRealtimeBuffer 缓冲音频。PR 版本里，async_chunk=true 时每凑够约 5 秒就 yield 一个 TokensPrompt(prompt_token_ids=..., multi_modal_data={"audio": segment})；async_chunk=false 时为避免 full-payload 路径重复响应，中间片不会 yield，只在 flush() 时发剩余音频。

4.2 engine 到 scheduler

AsyncOmni._add_streaming_input_request()
  first chunk:
    AsyncOmniEngine.add_request_async(..., resumable=True)
      -> StageSubmissionMessage(type="add_request")
      -> Orchestrator._handle_add_request()
  later chunks:
    AsyncOmniEngine.add_streaming_update_async(..., resumable=True)
      -> StageSubmissionMessage(type="streaming_update")

_validate_streaming_input_sampling_params() 当前限制：SamplingParams、n == 1、output_kind != FINAL_ONLY、不允许 stop strings。这个限制是为了让 session continuation 和 delta 输出状态机更简单。

5. async_chunk 时序

5.1 不开 async_chunk：Orchestrator 显式转发

Stage-0 Thinker segment done
  -> EngineCoreOutput(is_segment_finished=True)
  -> Orchestrator._route_output()
       if streaming.enabled and segment_finished:
         next_client.process_engine_inputs(..., streaming_context=req_state.streaming)
         next_pool.submit_initial/update(...)
  -> downstream stage sees normal request/update

这条路径的关键点是 streaming_context.new_prompt_len_snapshot。Stage-0 可能已经把下一片 prompt 追加进 request；thinker2talker() / thinker2talker_token_only() 会用这个 snapshot 去掉“下一片新 prompt tail”，避免把下一片输入错算进上一片 Talker 输入。

5.2 开 async_chunk：下游先 prewarm，数据靠 connector chunk

first streaming slice
  Orchestrator._handle_add_request()
    -> submit Stage-0 real request
    -> _prewarm_async_chunk_stages()
         Stage-1/2 submit placeholder prompt
         downstream request.resumable=True

Stage-0 executes
  OmniARScheduler.update_from_output()
    -> chunk_transfer_adapter.save_async(pooler_output, request, is_segment_finished)
    -> custom processor: thinker2talker_async_chunk()
    -> connector.put(external_req_id_stage_chunk_id, payload)

Stage-1 scheduler loop
  chunk_transfer_adapter.process_pending_chunks()
    -> request WAITING_FOR_CHUNK
    -> connector.get(...)
    -> request.additional_information = payload
    -> request.status = WAITING / RUNNING

current slice ends
  payload.meta.is_segment_finished = True
  downstream stops polling this segment

next input slice arrives
  Orchestrator._handle_streaming_update()
    -> Stage-0 streaming_update
    -> _prewarm_async_chunk_stages() again
  downstream scheduler clears segment_finished_requests
  next connector chunk resumes the same logical session
字段含义为什么需要
request.resumable这是一个 streaming session，当前 segment 结束后还可能继续。Request finish 后不要立刻整体释放；下游 prewarm request 也要知道后续能继续。
meta.is_segment_finished当前音频片结束。让下游停止等待本片更多 chunk，但不要清掉整个 request。
meta.finished整个 streaming request 结束。只有它为真，才走 terminal cleanup。
num_output_placeholdersvLLM async scheduling 预占但尚未确认的输出 token。PR 用 confirmed token cursor 发送 chunk，避免把未确认 token/embedding 传下游。

6. 代码走读主线

6.1 RealtimeConnection：输入启动和输出 delta

vllm_omni/entrypoints/openai/realtime_connection.py 继承上游 realtime connection。输入侧还是上游逻辑：append 解码 PCM16 入队，commit(final=false) 启动 generation，commit(final=true) 放入 None sentinel。Omni 覆写的重点在 _run_generation()：

  • 调用 self.engine.generate(prompt=streaming_input_gen, request_id=..., sampling_params_list=...)。
  • 只把 stage_id == 0 的 text delta 回传为 TranscriptionDelta，并把 Stage-0 新 token id 放入 input_stream，供下一轮 realtime prompt 构造上下文。
  • _extract_audio_chunks() 支持 multimodal_output["audio"] 或 ["model_outputs"]；_raw_waveform_to_deltas() 同时兼容“累计 waveform”和“真 delta waveform”。
  • 最后总是发 response.audio.done，避免客户端挂住。

6.2 AsyncOmni / AsyncOmniEngine：把 generator 变成 request update

AsyncOmni.generate() 检测 prompt 是 AsyncGenerator 后进入 _add_streaming_input_request()。第一片音频用 add_request_async(..., resumable=True)，后续片用 add_streaming_update_async(..., resumable=True)。generator 结束时，如果已经提交过首片，就发一个 TokensPrompt([0]) 且 resumable=False 的 final update。

AsyncOmniEngine._build_add_request_message() 负责把 prompt 经过 stage-0 input processor 变成 OmniEngineCoreRequest，并把 resumable 写进 request；StageSubmissionMessage.type 只有两个值：add_request 和 streaming_update。

6.3 Orchestrator：streaming state 与 stage 路由

  • OrchestratorRequestState.streaming 保存 enabled、segment_finished、new_prompt_len_snapshot、bridge_states。
  • _handle_add_request() 用 prompt.resumable 打开 streaming flag；PR #3614 让 async_chunk 模式在首片后调用 _prewarm_async_chunk_stages()。
  • _handle_streaming_update() 找到现有 request state 后调用 StagePool.submit_update()；PR #3614 新增：streaming update 时也会 prewarm downstream stages，并把 downstream request 标为 resumable。
  • _orchestration_loop() 从 raw EngineCoreOutput 把 is_segment_finished 和 new_prompt_len_snapshot 写回 request state。
  • _route_output() 在 non-async_chunk 路径下，遇到 finished 或 streaming segment finished 就 _forward_to_next_stage()；如果是 streaming session 且整条 request finished，会额外发一个 resumable=False terminal update 给下游。

6.4 Scheduler：session update 怎么改 Request

OmniARScheduler._update_request_as_session() 是 Stage-0 Thinker 的核心。它先记录新片 prompt 长度到 _new_prompt_len_snapshot，再处理 async scheduling 的未确认 placeholder：如果 num_output_placeholders > 0，设置 discard_latest_async_tokens=True，回退 num_computed_tokens，清空 spec tokens，避免把上一片末尾尚未确认的 token 带入下一片。

Stage-0 后续调用 upstream session update 逻辑，把“旧 prompt + 已确认输出 + 新片 prompt”拼成新的 prompt。下游 async_chunk stage 不在 update 时替换 prompt，因为真实 prompt/additional information 要等 connector chunk 到来；它只清掉 segment_finished_requests 并把状态放回 WAITING。

OmniGenerationScheduler._update_request_as_session() 是 Code2Wav/generation 类 stage 的替换语义：清空 _output_token_ids 和 _all_token_ids，把 update prompt 作为新 prompt，num_computed_tokens=0，并刷新 block hashes。

6.5 Connector / async_chunk：chunk 生产、接收、finish marker

OmniChunkTransferAdapter.save_async() 计算 confirmed watermark：num_computed_tokens - num_output_placeholders。这就是 PR 里“不能把 async scheduling 多产出的最后一个 placeholder embedding 传给下游”的落点。_send_single_request() 会把 custom processor 产出的 OmniPayloadStruct 送到 connector，同时强制写入两个 meta：

payload_data.meta.finished = is_finished
payload_data.meta.is_segment_finished = is_segment_finished

_poll_single_request() 接收 chunk 后，如果是 AR 下游，直接把 payload 放进 request.additional_information；如果 chunk_id > 0 and request.resumable，会调用 construct_next_stage_streaming_input_prompt()。这个 helper 不重置 num_computed_tokens，而是把已计算输出移动进 prompt_token_ids，再追加按 ids.prompt 算出的新 placeholder prompt，并刷新 block hashes。

6.6 Qwen3-Omni stage processors：Thinker -> Talker -> Code2Wav

thinker2talker_async_chunk() 在 chunk 0 处理首个 prefill，后续如果 request.resumable 走 _construct_thinker2talker_streaming_input_async_chunk()：

  • 当 thinker_emb.shape[0] > 1，说明新 streaming input segment 带来了 prefill embeddings/hidden states。它先缓存到 _pending_streaming_prefills[request_id]，返回 None，不立刻发下游。
  • 下一次 decode chunk 到来时，把缓存的 prefill 与当前 decode embedding 拼起来发送，保证 Talker 的 ids 和 tensor rows 对齐。
  • 没有 output token 且未 finish 时返回 None，避免把 async placeholder 或 stop token 的 embedding/hidden state 发下游。

_get_streaming_talker_tokens() 用 last_prompt_len / last_output_len 只取本 segment 增量；_get_streaming_codec_delta_len() 用 talker2code2wav_last_seq_len 只取新 codec 帧。talker2code2wav_async_chunk() 再按 codec_chunk_frames 聚合 codec rows，并携带 left_context_size 供 Code2Wav 做流式解码。

6.7 Model runner：流式 prefill 的 additional info

OmniGPUModelRunner 遇到同一个 req_id 再次出现在 scheduled_new_reqs 时，走 _update_streaming_request()；PR #3614 把 _update_streaming_input_additional_info() 简化成只设置 meta.num_processed_tokens = 0 和 meta.resumable = True。Qwen3 Talker 的 talker_preprocess() 看到 meta.resumable 时，不再用 runner 的 num_computed_tokens 覆盖 meta.num_processed_tokens，避免 streaming prefill 被错误当成已处理过。

7. 类级代码拆解

下面按真实调用链拆类。读的时候注意两条边界：HTTP/WebSocket 入口只负责把输入变成 StreamingInput，engine/orchestrator/scheduler 才负责 request 生命周期；async_chunk 的 connector 只传 stage 间 payload，不负责决定用户级 request 是否结束。

7.1 上游 RealtimeConnection：WebSocket 会话状态机

文件vllm/entrypoints/speech_to_text/realtime/connection.py
类职责管理 websocket lifecycle、session validation、音频 queue、generation task。它是协议层，不知道 Omni 多 stage，也不知道 async_chunk。
关键字段audio_queue 存 PCM float32 chunk 或 None sentinel；generation_task 防止同一连接重复启动 generation；_is_model_validated 要求先 session.update。
关键方法handle_event() 分发 session.update / input_audio_buffer.append / input_audio_buffer.commit；start_generation() 创建 audio_stream_generator() 和 input_stream；_run_generation() 上游默认只发 transcription text。
输入输出输入是 websocket JSON 和 base64 PCM16；输出是 TranscriptionDelta / TranscriptionDone。Omni 子类扩展 audio delta。

7.2 Omni RealtimeConnection：把 Omni 输出转成 text/audio delta

文件vllm_omni/entrypoints/openai/realtime_connection.py
继承关系继承上游 VllmRealtimeConnection，入口事件仍复用上游，主要重写 _run_generation() 和 audio 提取逻辑。

7.3 OpenAIServingRealtime：音频流到 StreamingInput

文件上游 vllm/entrypoints/speech_to_text/realtime/serving.py
类职责把 websocket 层的 audio_stream 变成 engine 可消费的 AsyncGenerator[StreamingInput]。
关键方法transcribe_realtime() 调用模型类的 buffer_realtime_audio(audio_stream, input_stream, model_config)，再用 renderer 把模型 prompt 渲染为 engine input，最后 yield StreamingInput(prompt=engine_input)。
边界这里不切音频、不做 session continuation；切片策略在模型类 buffer_realtime_audio 里。

7.4 Qwen3OmniMoeForConditionalGeneration：模型级切片与 Talker 预处理

文件vllm_omni/model_executor/models/qwen3_omni/qwen3_omni.py
类职责同一个 wrapper 根据 model_stage 承载 Thinker、Talker、Code2Wav。对 realtime input，它提供类方法 buffer_realtime_audio()。
buffer_realtime_audio()创建 Qwen3ASRRealtimeBuffer(segment_duration_s=5.0)；构造固定 chat 模板 <|im_start|>user\n{audio_placeholder}...；async_chunk=true 时每读到完整 segment 就 yield 一个带 multi_modal_data={"audio": segment} 的 TokensPrompt；最后 flush() 剩余音频。
Qwen3ASRRealtimeBuffer音频切片状态机。append() 累积输入波形；达到 segment_duration_s 后吐出一个 segment；flush() 在 final sentinel 后吐出剩余尾巴。这里的 segment 边界就是后面 AsyncOmni 第一片 add_request、后续片 streaming_update 的来源。
为什么看 async_chunkPR 版本里 async_chunk=false 不在中间 yield segment，避免 full-payload 模式把每片当 fresh decode 造成重复回答/重复音频。
talker_preprocess()Talker prefill 时会参考 payload meta。PR 加了 request_resumable = meta.get("resumable", False)，resumable 时不让 runner 的 num_computed_tokens 覆盖 meta.num_processed_tokens。

7.5 AsyncOmni：用户态 generate 协调器

文件vllm_omni/entrypoints/async_omni.py
类职责对外提供统一 generate() API，隐藏多 stage orchestrator。它拥有 request 级 ClientRequestState，负责把 final output queue 里的消息 yield 给调用者。
generate()先解析 sampling params、final stage、metrics；如果 prompt 是 AsyncGenerator，进入 streaming input 分支，否则普通 add_request_async()。
_add_streaming_input_request()后台 task 消费 input_stream。第一片调用 engine.add_request_async(..., resumable=True)，后续片调用 engine.add_streaming_update_async(..., resumable=True)。finally 中发 final sentinel：已有首片则 streaming_update(resumable=False)，否则 add_request(resumable=False)。
约束_validate_streaming_input_sampling_params() 限制 n == 1、非 FINAL_ONLY、无 stop strings，保证 delta 和 continuation 状态机可控。

7.6 AsyncOmniEngine 与 StageSubmissionMessage：入队消息格式

文件vllm_omni/engine/async_omni_engine.py，vllm_omni/engine/messages.py
StageSubmissionMessagemsgspec struct，type 只有 add_request 或 streaming_update；携带 request_id、预处理后的 prompt、原始 original_prompt、sampling_params_list、final_stage_id、耗时字段。
_build_add_request_message()做 stage-0 input processing：注入 global id，scope 多模态 cache 到 stage-0 replica，调用 input_processor.process_inputs(..., resumable=resumable)，再升级为 OmniRequest 并写入 external_req_id。
add_request_async()同步构造 StageSubmissionMessage(type="add_request")，放入 orchestrator 的 request queue。
add_streaming_update_async()构造同样的 message，但 message_type="streaming_update"。这让 Orchestrator 能区别“新 session”与“已有 session continuation”。

7.7 OrchestratorRequestState / StreamingInputState：Orchestrator 侧会话记忆

文件vllm_omni/engine/orchestrator.py
OrchestratorRequestState每个 logical request 一份。保存原始 prompt、每 stage sampling params、final stage、stage submit 时间、mm features、PD prefill output、pipeline timings，以及 streaming 状态。
StreamingInputStateenabled 表示这是 streaming input；segment_finished 表示当前输入片已结束；new_prompt_len_snapshot 记录 Stage-0 新片 prompt 长度；bridge_states 给 Qwen3 processor 存跨 segment 的游标。
关键点这是 Orchestrator 级别的“事实来源”。Scheduler 产出的 EngineCoreOutput.is_segment_finished 和 new_prompt_len_snapshot 会被写回这里，下游 process_engine_inputs(..., streaming_context=req_state.streaming) 再读取。

7.8 Orchestrator：跨 stage request 状态机

文件vllm_omni/engine/orchestrator.py
_handle_add_request()创建 OrchestratorRequestState，用 prompt.resumable 设置 streaming.enabled，提交 Stage-0。PR #3614 在 async_chunk 且存在下游 stage 时调用 _prewarm_async_chunk_stages()。
_handle_streaming_update()查已有 request state，更新 sampling params，标记 streaming enabled，调用 Stage-0 submit_update()。PR #3614 也在这里 prewarm downstream，让下一片可以继续同一条 async_chunk 管线。
_orchestration_loop()轮询 stage raw output。对每个 raw EngineCoreOutput，把 is_segment_finished 和 new_prompt_len_snapshot 同步到 req_state.streaming。
_route_output()如果当前 stage 是 final output，发给前端 queue；如果不是 final 且非 async_chunk，遇到 finished 或 streaming segment finished 时显式 _forward_to_next_stage()。async_chunk 模式不走这条显式转发，数据由 connector 传。
_forward_to_next_stage()non-async_chunk 的核心桥：调用下一 stage client 的 process_engine_inputs()，把 source output 转成下一 stage prompt；streaming session 的中间 update 用 resumable=True，terminal update 用 resumable=False。
_prewarm_async_chunk_stages()async_chunk 专用：给 Stage-1/Stage-2 提前提交 placeholder prompt。Talker prompt 长度用 compute_talker_prompt_ids_length(prompt_token_ids) 估计；PR #3614 把 downstream request 的 resumable 设为 stage-0 request 的 resumable。

7.19 补充数据类与协议类：request、output、stage client 边界

这些类不一定承载复杂算法，但它们定义了跨模块边界。流式问题最容易出错的地方，通常就是这些结构里的 flag、id 和 payload 没有被完整透传。

类/结构文件代码拆解
PromptEmbedsPayload vllm_omni/engine/__init__.py 把 prompt embedding 序列化成 data: bytes，并保存 shape 与 dtype。shape 语义是 [seq_len, hidden_size]：seq_len 对齐 prompt token/placeholder 行数，hidden_size 对齐当前 stage 模型隐层维度。它解决的是跨进程传 prompt embeddings 不能直接传 tensor 引用的问题。
AdditionalInformationPayload vllm_omni/engine/__init__.py 把 stage 间附加信息打成可序列化 dict，value 是 AdditionalInformationEntry。在 async_chunk 下，真正的 tensor payload 多数由 connector 传；这个结构更多承载小控制字段、shape、meta、语言/说话人等能随 request 进入 scheduler 的信息。
OmniEngineCoreRequest vllm_omni/engine/__init__.py 继承/扩展 vLLM EngineCoreRequest，增加 prompt_embeds、additional_information、external_req_id。from_request() 会复制 upstream request 的通用字段，同时保留 resumable。这一步很关键：如果 resumable 在 engine core request 层丢失，scheduler 就无法把 segment finish 和 request finish 区分开。
OmniEngineCoreOutput vllm_omni/engine/__init__.py 扩展 vLLM EngineCoreOutput，增加 pooling_output、is_segment_finished、new_prompt_len_snapshot。pooling_output 是 stage processor/connector 的原料；is_segment_finished 告诉 Orchestrator “本片结束但 request 可能未结束”；new_prompt_len_snapshot 给 non-async_chunk 转发路径切分本 segment 的 delta。
OmniEngineCoreOutputs vllm_omni/engine/__init__.py 批量 output 容器，语义上对应一次 engine step 返回的多个 OmniEngineCoreOutput。它没有复杂逻辑，但保证 Orchestrator 能按 stage poll 到同一批结构化输出，而不是混用 upstream output 类型。
StageClient vllm_omni/engine/stage_client.py 协议类，定义 Orchestrator 能对任意 stage client 做什么：add_request_async、add_streaming_update_async、abort_request_async、step_async、process_engine_inputs。它把“stage 是本地 engine 还是远端 replica”的差异挡在 Orchestrator 外面。
StagePoolLLMClient vllm_omni/engine/stage_client.py 实际 LLM stage client 封装，负责把 Orchestrator 传入的 prompt/sampling params 转给底层 engine client。它也暴露 process_engine_inputs()，用于 non-async_chunk 显式转发时把上一 stage 的 output 转成下一 stage input。
OmniModelRunnerOutput vllm_omni/outputs.py runner 到 scheduler 的执行结果容器，扩展点是 omni_connector_output。模型 forward 的 token/logits 等仍走 vLLM 原有路径；connector 的 chunk ready、chunk finished、kv transfer 等控制信号通过这个字段旁路给 scheduler。
OmniRequestOutput vllm_omni/outputs.py 最终返回给 AsyncOmni.generate() 调用者的统一输出结构。它包含 stage_id、finished、request_output、images、latents、metrics、_multimodal_output、error。from_pipeline() 包装 AR/pipeline 输出，from_diffusion() 包装 Code2Wav/diffusion 输出，from_error() 让异常也能沿同一条 async generator 交付。

L802-L824talker2code2wav_async_chunk 先验证 pooling output 里有 codes.audio，再从 connector config 读取 codec_chunk_frames 和 codec_left_context_frames。这两个参数决定多少 codec frame 聚成一次 vocoder 输入，以及左上下文带多少。
L826-L854过滤空 codes、全 0 codes、stop token codec frame。这里防止 terminal token 生成伪音频，也防止空 chunk 让 Code2Wav 空跑。
L856-L877把 codec codes 转成扁平 list，按 request 累积；未达到 chunk size 且未 finished 就返回 None。到达边界后计算 left context，取最近窗口，转成 CodesStruct(audio=...)，并在 meta 里写 left_context_size 和 finished。
