Vime 框架导读：从 train.py 到 vLLM Rollout

目标：按代码路径解释 Vime 的训练闭环、每一步数据如何变化、它如何把 vLLM 接进 slime，以及 Megatron 在这里承担什么角色。

代码基于本地 vime/。Vime 是 slime fork，当前重点是把 rollout 后端切到 vLLM + vllm-router，仍保留不少 slime / sglang 命名和兼容层。

1. 先看怎么用

Vime 的入口仍然很直接：用户准备 Megatron 格式训练 checkpoint、HF checkpoint/tokenizer、prompt 数据，然后运行 python train.py ...。关键参数可以按四组理解：

参数组典型参数作用
资源分配--actor-num-nodes、--actor-num-gpus-per-node、--rollout-num-gpus、--rollout-num-gpus-per-engine、--colocateRay placement group 里哪些 GPU 给 Megatron 训练，哪些 GPU 给 vLLM rollout。rollout-num-gpus-per-engine 基本等价每个 vLLM engine 的 tensor parallel size。
模型与 checkpoint--hf-checkpoint、--load、--ref-load、--savehf-checkpoint 给 vLLM / tokenizer / processor 用；训练权重来自 Megatron checkpoint，并在启动后同步给 vLLM。
rollout 与数据--prompt-data、--input-key、--label-key、--rollout-batch-size、--n-samples-per-prompt每轮取多少 prompt、每个 prompt 采几条 response、如何算 reward。
RL 与训练--advantage-estimator、--global-batch-size、--num-steps-per-rollout、Megatron 并行参数控制 GRPO / GSPO / PPO / REINFORCE++，以及 Megatron TP/PP/CP/EP、micro-batch、动态 batch 等。
一个最重要的约束：rollout_batch_size * n_samples_per_prompt 是一轮 rollout 产生的样本数；global_batch_size * num_steps_per_rollout 是一轮训练消费的样本数。默认 on-policy 训练里两边需要对齐。

2. 整体架构

Vime 的核心不是单个 trainer，而是一个由 Ray 编排的训推闭环：

模块代码路径职责
入口vime/train.py创建资源、启动 rollout、启动训练 actor、循环执行 generate/train/save/eval/weight sync。
Ray 编排slime/ray/placement_group.py、rollout.py、actor_group.py固定 GPU 排布，创建 Ray actor，把 rollout server 和训练进程连起来。
vLLM 后端slime/backends/vllm_utils/、slime/rollout/vllm_rollout.py启动 vllm serve、注册 router、发 HTTP 请求、解析 token/logprob/routed experts。
训练后端slime/backends/megatron_utils/Megatron 初始化、数据 packing、logprob/value/advantage/loss、checkpoint、权重同步。
数据抽象slime/utils/data.py、slime/utils/types.py、slime/rollout/data_source.py把 jsonl/parquet 行转成 Sample，再把 rollout 后的 Sample 转成训练 batch。

3. train.py 主流程图

train.py 的逻辑很薄，但它定义了整个系统的时序。可以把一轮训练看成：生成新数据、训练 Megatron、把新权重推回 vLLM。

# vime/train.py 的主循环骨架
rollout_data_ref = ray.get(rollout_manager.generate.remote(rollout_id))

if args.use_critic:
    value_refs = critic_model.async_train(rollout_id, rollout_data_ref)
    ray.get(actor_model.async_train(rollout_id, rollout_data_ref, external_data=value_refs))
else:
    ray.get(actor_model.async_train(rollout_id, rollout_data_ref))

actor_model.update_weights()
ray.get(rollout_manager.eval.remote(rollout_id))  # 按 interval 触发

4. 训练数据如何流动

Vime 的数据单位是 Sample。它从 prompt 开始，经过 vLLM 生成后带上 response、tokens、reward、logprobs，最后被 RolloutManager 转成 Megatron 可消费的 dict。

4.1 Sample 字段

阶段关键字段来源
prompt 初始化prompt、label、metadata、multimodal_inputsslime/utils/data.py::Dataset 读取 jsonl/parquet，必要时调用 tokenizer chat template 和 processor。
分组采样group_index、index、每个 prompt 复制 n_samples_per_prompt 份RolloutDataSource.get_samples。
vLLM 生成tokens、response、response_length、status、rollout_log_probsvllm_rollout.generate 调 /inference/v1/generate 后解析 choices[0]。
rewardrewardasync_rm / batched_async_rm / 自定义 reward；group RM 可在整组生成后统一算。
训练附加loss_mask、teacher_log_probs、rollout_routed_expertspartial rollout、OPD、MoE routing replay 等特性按需写入。

4.2 train_data 字典

RolloutManager._convert_samples_to_train_data 默认产物大概是：

{
  "tokens": [[prompt_ids..., response_ids...], ...],
  "response_lengths": [R0, R1, ...],
  "rewards": [advantage_input_reward0, ...],
  "raw_reward": [raw_reward0, ...],
  "truncated": [0_or_1, ...],
  "sample_indices": [sample.index, ...],
  "loss_masks": [[1, 1, 0, ...], ...],
  "rollout_log_probs": [[lp_for_response_token...], ...],      # 可选
  "rollout_routed_experts": [np.ndarray[R-1, layers, topk], ...] # 可选
}

之后 _split_train_data_by_dp 会按 data parallel size 拆成多个 Ray object。Megatron 每个 DP rank 只拿自己那份，避免所有训练 rank 都处理全量数据。

4.3 Megatron 侧 packing 后的 shape

字段进入 Megatron 前get_batch 后含义
tokens每个 sample 一个 1D tensor，长度 prompt_len + response_lenthd: [1, T_padded]；bshd: [B, S_padded]Vime 默认强调 data packing，thd 下 T 是 micro-batch 内 token 拼接后的长度。
full_loss_masks每个 sample 长度为 response_len与 tokens 同 shapeprompt 区域和最后一个位置补 0，只在 response token 上算 loss。
packed_seq_params无PackedSeqParams(cu_seqlens_q, max_seqlen_q, qkv_format="thd")告诉 Megatron/Transformer Engine 每条序列的边界。

5. Vime 如何与 vLLM 交互

Vime 和 vLLM 的边界非常清楚：Vime 用 Ray actor 管 vLLM 进程生命周期，用 HTTP 调 vLLM server 的控制面和推理面。

5.1 vLLM server 是怎么启动的

ServerGroup.start_engines 创建 VLLMEngine Ray actor；VLLMEngine._init_normal 再用 Python multiprocessing 执行 vllm serve。启动命令由 launch_server_process 组装：

vllm serve $hf_checkpoint \
  --tensor-parallel-size $rollout_num_gpus_per_engine \
  --port $server_port \
  --host $host \
  --seed $seed \
  --trust-remote-code \
  --gpu-memory-utilization 0.55 \
  --weight-transfer-config '{"backend":"ipc" or "nccl"}'

Vime 会把 vLLM 原生 AsyncEngineArgs 自动加前缀暴露成 --vllm-*，再在启动 subprocess 时转回 vLLM CLI。比如 --vllm-max-model-len 会最终进入 vllm serve --max-model-len。

5.2 推理请求用的不是 OpenAI Chat Completions

默认文本路径走 vLLM 的 token-only 接口：

POST http://{router_ip}:{router_port}/inference/v1/generate
{
  "model": args.hf_checkpoint,
  "token_ids": [prompt_token_ids...],
  "sampling_params": {
    "max_tokens": args.rollout_max_response_len,
    "temperature": args.rollout_temperature,
    "top_p": args.rollout_top_p,
    "logprobs": 1
  }
}

这样做的好处是避免生成后再重新 tokenize：vLLM 返回 choice.token_ids 和 logprobs.content，Vime 直接把新 token append 到 sample.tokens，同时记录 rollout-time logprob。

5.3 多模态请求

如果 sample.multimodal_inputs["images"] 存在，Vime 先调用 /v1/chat/completions/render 做多模态预处理，拿到 token/features，再转成 /inference/v1/generate 的 body。这对应 vllm_rollout.py::_mm_render_response_to_generate_body。

6. 训练权重如何同步给 vLLM

每轮 actor 训练完，train.py 调 actor_model.update_weights()。真正执行的是每个 Megatron rank 上的 MegatronTrainRayActor.update_weights。

模式代码大致流程
训推分离update_weight_from_distributed.pyMegatron PP 源 rank 组织 NCCL group，必要时 all-gather TP 参数，转换到 HF 名称，调用 vLLM /init_weight_transfer_engine、/start_weight_update、/update_weights、/finish_weight_update。
训推同卡 colocateupdate_weight_from_tensor.py先让 vLLM /sleep?level=0 释放 weights/KV，再通过 CUDA IPC 把同 GPU slot 的 tensor handle 传给 vLLM worker extension。
外部 vLLM--rollout-externalVime 不启动 server，只连接用户给的地址，但仍会做健康检查和权重同步相关控制。

7. Megatron 简明科普

如果只熟悉 PyTorch / HF Trainer，可以先把 Megatron 理解成“大模型训练专用运行时”。它主要解决一个问题：单卡放不下、算不快时，如何把模型、序列、batch、专家切到很多 GPU 上，并且让 forward/backward/optimizer 仍然正确。

7.1 常见并行名词

缩写含义直觉
DPData Parallel每组 GPU 拿不同样本，梯度 all-reduce。
TPTensor Parallel把一层里的大矩阵切到多张卡。
PPPipeline Parallel把不同 transformer layers 放到不同卡，像流水线一样跑。
CPContext Parallel把长序列维度切开，常和 ring attention 相关。
EPExpert ParallelMoE expert 分布到不同卡，token 按路由去对应 expert。
SPSequence ParallelTP 的配套优化，进一步切部分 sequence 激活。

7.2 Vime 里 Megatron 做什么

  • 初始化 actor / critic / ref / teacher 模型。
  • 加载和保存 Megatron checkpoint，必要时导出 HF checkpoint。
  • 把 rollout 样本 packing 成 Megatron forward 需要的 batch。
  • 计算当前 actor logprob、ref logprob、critic values。
  • 计算 GRPO / GSPO / PPO / REINFORCE++ advantage 和 returns。
  • 执行 policy loss / value loss / SFT loss 的 forward-backward 和 optimizer step。
  • 把训练后的 actor 权重转成 vLLM 能接收的 HF 命名并同步过去。

7.3 一条样本在 Megatron 训练时怎么变

# rollout 后一条样本
tokens = [prompt_token_ids..., response_token_ids...]   # 长度 total_length
response_length = R
loss_mask = [1, 1, ..., 1]                              # 长度 R
reward = 1.0

# get_batch 对齐到 token-level loss
full_loss_mask = [0] * (prompt_length - 1) + loss_mask + [0]

# actor forward 得到 logits [1, T_padded, vocab]
# get_log_probs_and_entropy 从 logits 里只切 response 对应位置：
log_probs = log p_actor(response_t | prompt, response_<t)

# ref / rollout logprobs 和 reward 进入 advantage：
advantages, returns = compute_advantages_and_returns(...)

# policy_loss_function 计算 PPO-style clipped loss：
ppo_kl = old_log_probs - new_log_probs
pg_loss = clipped_policy_gradient(ppo_kl, advantages)

这里容易混淆的是：rollout_log_probs 是 vLLM 生成时记录的旧策略概率；log_probs 是 Megatron 当前 actor 对同一批 response 重算出来的概率。off-policy correction、mismatch metrics、TIS 都围绕两者的差异展开。

8. Vime / slime 特性清单

特性入口参数 / 代码说明
vLLM rollout 后端slime/backends/vllm_utils、slime/rollout/vllm_rollout.py默认 rollout function 已改为 vLLM；vLLM 原生参数通过 --vllm-* 透传。
vllm-router--router-*、_start_routerVime 启动 router，vLLM workers 注册到 router，生成请求统一打到 router。
训推分离 / 训推同卡--colocate、--offload-train、--offload-rollout分离模式用 NCCL 同步权重；同卡模式配合 sleep/wake 和 CUDA IPC 节省显存。
动态采样过滤--dynamic-sampling-filter-path类似 DAPO，可丢弃全对/全错等没有训练价值的 group，继续 oversampling。
partial rollout--partial-rollout、--mask-offpolicy-in-partial-rollout长回答未完成时可 abort 并回收到 buffer；续写时可 mask 掉旧 token，只训练本轮 on-policy token。
rollout bufferRolloutDataSourceWithBuffer、slime_plugins/rollout_bufferDataSource 可从 prompt dataset 和 buffer 混合取样。
多算法--advantage-estimator支持 grpo、gspo、ppo、reinforce_plus_plus、reinforce_plus_plus_baseline。
OPD 蒸馏--use-opd、teacher_log_probs可用 teacher logprob 对 advantage 加 reverse KL 惩罚。
MoE routing replay--use-rollout-routing-replayvLLM 返回 routed experts，Megatron 训练时复用路由，降低 train/infer mismatch。
动态 batch size--use-dynamic-batch-size、--max-tokens-per-gpu根据样本 token 长度决定 micro-batch 数，并做长度均衡 packing。
容错--use-fault-tolerance健康监控发现 rollout engine 挂掉后重启，并在下次 weight sync 补权重。
注意：当前 Vime 仍在开发中，很多接口名沿用 slime / sglang，例如 SglangConfig、sglang_model_routers、--sglang-config。阅读时应看实际实现：默认 engine 已经是 VLLMEngine，默认 rollout function 已经是 slime.rollout.vllm_rollout.generate_rollout。

10. 代码核对清单

问题看哪里结论
Vime 的默认 rollout 后端是不是 vLLM？slime/utils/arguments.py 的 --rollout-function-path默认是 slime.rollout.vllm_rollout.generate_rollout。
vLLM server 是外部服务还是 Vime 自己拉起？VLLMEngine._init_normal默认由 Vime spawn vllm serve；--rollout-external 时连接外部地址。
请求发到哪个接口？vllm_rollout.generate文本默认 /inference/v1/generate；多模态先 /v1/chat/completions/render。
训练数据在哪里从 Sample 变成 dict？RolloutManager._convert_samples_to_train_data产出 tokens/response_lengths/rewards/loss_masks/rollout_log_probs 等字段。
dict 如何分给 Megatron DP rank？RolloutManager._split_train_data_by_dp按 DP 数切 partition，可按 sequence length balance。
Megatron 哪里把 list token pack 成 batch？megatron_utils/data.py::get_batchthd 模式拼成 [1, T_padded] 并构造 PackedSeqParams。
GRPO/PPO advantage 在哪里算？megatron_utils/loss.py::compute_advantages_and_returns按 args.advantage_estimator 分支。
权重同步给 vLLM 在哪里？MegatronTrainRayActor.update_weightscolocate 走 UpdateWeightFromTensor，分离走 UpdateWeightFromDistributed。
