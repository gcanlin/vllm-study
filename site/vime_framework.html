1. 先看怎么用

Vime 的入口仍然很直接：用户准备 Megatron 格式训练 checkpoint、HF checkpoint/tokenizer、prompt 数据，然后运行 python train.py ...。关键参数可以按四组理解：

参数组典型参数作用
资源分配--actor-num-nodes、--actor-num-gpus-per-node、--rollout-num-gpus、--rollout-num-gpus-per-engine、--colocateRay placement group 里哪些 GPU 给 Megatron 训练，哪些 GPU 给 vLLM rollout。rollout-num-gpus-per-engine 基本等价每个 vLLM engine 的 tensor parallel size。
模型与 checkpoint--hf-checkpoint、--load、--ref-load、--savehf-checkpoint 给 vLLM / tokenizer / processor 用；训练权重来自 Megatron checkpoint，并在启动后同步给 vLLM。
rollout 与数据--prompt-data、--input-key、--label-key、--rollout-batch-size、--n-samples-per-prompt每轮取多少 prompt、每个 prompt 采几条 response、如何算 reward。
RL 与训练--advantage-estimator、--global-batch-size、--num-steps-per-rollout、Megatron 并行参数控制 GRPO / GSPO / PPO / REINFORCE++，以及 Megatron TP/PP/CP/EP、micro-batch、动态 batch 等。
一个最重要的约束：rollout_batch_size * n_samples_per_prompt 是一轮 rollout 产生的样本数；global_batch_size * num_steps_per_rollout 是一轮训练消费的样本数。默认 on-policy 训练里两边需要对齐。

2. 整体架构

Vime 的核心不是单个 trainer，而是一个由 Ray 编排的训推闭环：

模块代码路径职责
入口vime/train.py创建资源、启动 rollout、启动训练 actor、循环执行 generate/train/save/eval/weight sync。
Ray 编排slime/ray/placement_group.py、rollout.py、actor_group.py固定 GPU 排布，创建 Ray actor，把 rollout server 和训练进程连起来。
vLLM 后端slime/backends/vllm_utils/、slime/rollout/vllm_rollout.py启动 vllm serve、注册 router、发 HTTP 请求、解析 token/logprob/routed experts。
训练后端slime/backends/megatron_utils/Megatron 初始化、数据 packing、logprob/value/advantage/loss、checkpoint、权重同步。
数据抽象slime/utils/data.py、slime/utils/types.py、slime/rollout/data_source.py把 jsonl/parquet 行转成 Sample，再把 rollout 后的 Sample 转成训练 batch。

3. train.py 主流程图

train.py 的逻辑很薄，但它定义了整个系统的时序。可以把一轮训练看成：生成新数据、训练 Megatron、把新权重推回 vLLM。

# vime/train.py 的主循环骨架

    ray.get(actor_model.async_train(rollout_id, rollout_data_ref))

actor_model.update_weights()
ray.get(rollout_manager.eval.remote(rollout_id))  # 按 interval 触发

5. Vime 如何与 vLLM 交互

Vime 和 vLLM 的边界非常清楚：Vime 用 Ray actor 管 vLLM 进程生命周期，用 HTTP 调 vLLM server 的控制面和推理面。

5.1 vLLM server 是怎么启动的

ServerGroup.start_engines 创建 VLLMEngine Ray actor；VLLMEngine._init_normal 再用 Python multiprocessing 执行 vllm serve。启动命令由 launch_server_process 组装：

vllm serve $hf_checkpoint \
  --tensor-parallel-size $rollout_num_gpus_per_engine \
  --port $server_port \
  --host $host \
  --seed $seed \
  --trust-remote-code \
  --gpu-memory-utilization 0.55 \
  --weight-transfer-config '{"backend":"ipc" or "nccl"}'

Vime 会把 vLLM 原生 AsyncEngineArgs 自动加前缀暴露成 --vllm-*，再在启动 subprocess 时转回 vLLM CLI。比如 --vllm-max-model-len 会最终进入 vllm serve --max-model-len。

5.2 推理请求用的不是 OpenAI Chat Completions

默认文本路径走 vLLM 的 token-only 接口：

POST http://{router_ip}:{router_port}/inference/v1/generate
{
  "model": args.hf_checkpoint,
  "token_ids": [prompt_token_ids...],
  "sampling_params": {
    "max_tokens": args.rollout_max_response_len,
    "temperature": args.rollout_temperature,
    "top_p": args.rollout_top_p,
    "logprobs": 1
  }
}

这样做的好处是避免生成后再重新 tokenize：vLLM 返回 choice.token_ids 和 logprobs.content，Vime 直接把新 token append 到 sample.tokens，同时记录 rollout-time logprob。

5.3 多模态请求

如果 sample.multimodal_inputs["images"] 存在，Vime 先调用 /v1/chat/completions/render 做多模态预处理，拿到 token/features，再转成 /inference/v1/generate 的 body。这对应 vllm_rollout.py::_mm_render_response_to_generate_body。

6. 训练权重如何同步给 vLLM

每轮 actor 训练完，train.py 调 actor_model.update_weights()。真正执行的是每个 Megatron rank 上的 MegatronTrainRayActor.update_weights。

模式代码大致流程

7. Megatron 简明科普

如果只熟悉 PyTorch / HF Trainer，可以先把 Megatron 理解成“大模型训练专用运行时”。它主要解决一个问题：单卡放不下、算不快时，如何把模型、序列、batch、专家切到很多 GPU 上，并且让 forward/backward/optimizer 仍然正确。

7.1 常见并行名词

缩写含义直觉
DPData Parallel每组 GPU 拿不同样本，梯度 all-reduce。
TPTensor Parallel把一层里的大矩阵切到多张卡。
PPPipeline Parallel把不同 transformer layers 放到不同卡，像流水线一样跑。
CPContext Parallel把长序列维度切开，常和 ring attention 相关。
EPExpert ParallelMoE expert 分布到不同卡，token 按路由去对应 expert。
SPSequence ParallelTP 的配套优化，进一步切部分 sequence 激活。

7.2 Vime 里 Megatron 做什么

  • 初始化 actor / critic / ref / teacher 模型。
  • 加载和保存 Megatron checkpoint，必要时导出 HF checkpoint。
  • 把 rollout 样本 packing 成 Megatron forward 需要的 batch。
  • 计算当前 actor logprob、ref logprob、critic values。
  • 计算 GRPO / GSPO / PPO / REINFORCE++ advantage 和 returns。
  • 执行 policy loss / value loss / SFT loss 的 forward-backward 和 optimizer step。
  • 把训练后的 actor 权重转成 vLLM 能接收的 HF 命名并同步过去。

7.3 一条样本在 Megatron 训练时怎么变

# rollout 后一条样本
tokens = [prompt_token_ids..., response_token_ids...]   # 长度 total_length
response_length = R
loss_mask = [1, 1, ..., 1]                              # 长度 R
reward = 1.0

# get_batch 对齐到 token-level loss
full_loss_mask = [0] * (prompt_length - 1) + loss_mask + [0]

# actor forward 得到 logits [1, T_padded, vocab]
# get_log_probs_and_entropy 从 logits 里只切 response 对应位置：
log_probs = log p_actor(response_t | prompt, response_<t)

# ref / rollout logprobs 和 reward 进入 advantage：
advantages, returns = compute_advantages_and_returns(...)

# policy_loss_function 计算 PPO-style clipped loss：
ppo_kl = old_log_probs - new_log_probs
pg_loss = clipped_policy_gradient(ppo_kl, advantages)

这里容易混淆的是：rollout_log_probs 是 vLLM 生成时记录的旧策略概率；log_probs 是 Megatron 当前 actor 对同一批 response 重算出来的概率。off-policy correction、mismatch metrics、TIS 都围绕两者的差异展开。

8. GRPO / PPO / reward / logprob

Vime 的 RL 训练可以先用一句话概括：vLLM 用旧 actor 采样 response，reward model 或规则给每条 response 打分，Megatron 用这些 response 重新计算当前 actor 的 logprob，再按 GRPO/PPO 等算法把 reward 转成 token-level advantage，最后更新 actor。

8.1 这些量分别是什么

量直觉Vime 代码位置训练里怎么用
reward一条 response 的最终分数，数学题常是对/错，代码题常是测试通过率，也可以是 RM 输出。vllm_rollout.generate_and_rm、rm_hub、RolloutManager._post_process_rewards先进入 train_data["rewards"]。GRPO 会按组归一化；PPO 会把它加到最后一个有效 response token 上。
raw_reward未归一化、未后处理的原始分数。_convert_samples_to_train_data主要用于日志、pass@k、debug。训练常用的是后处理后的 rewards。
rollout_log_probsvLLM 在生成时，旧 actor 对每个采样出来的 token 给出的 log probability。vllm_rollout.generate作为 old policy。off-policy correction、TIS、mismatch metrics 和部分 PPO ratio 会用。
log_probs训练时 Megatron 当前 actor 对同一批 response token 重新算的 log probability。MegatronTrainRayActor.compute_log_prob、loss.py::get_log_probs_and_entropy作为 new policy。policy loss 里的 ratio 本质是 exp(log_probs - old_log_probs)。
ref_log_probsreference model 对同一批 response 的 log probability。with_ref=True 后 compute_log_prob(store_prefix="ref_")计算 KL penalty 或 KL loss，防止 actor 偏离初始/参考模型太远。
valuescritic 对每个 response token 的价值估计，近似“从这里往后还能拿多少回报”。train_critic、loss.py::get_valuesPPO 用它做 GAE advantage 和 value loss；GRPO 默认不需要 critic。
advantages告诉 actor 某个 token 动作比预期好还是差。正数提高概率，负数降低概率。loss.py::compute_advantages_and_returnspolicy loss 的权重。它不是日志指标本身，而是训练信号。
returnscritic 要拟合的目标回报。compute_advantages_and_returnsPPO critic 的 value loss 目标；GRPO 中常和 reward broadcast 近似一致。

8.2 GRPO 怎么运作

GRPO 不训练 critic。它依赖“同一个 prompt 采多条 response”形成组内比较：同组里 reward 高的 response 得到正 advantage，低的 response 得到负 advantage。Vime 里这一步分两段完成。

# RolloutManager._post_process_rewards
raw_rewards = [sample.get_reward_value(args) for sample in samples]
rewards = rewards.reshape(-1, n_samples_per_prompt)
rewards = rewards - rewards.mean(dim=-1, keepdim=True)
if grpo_std_normalization:
    rewards = rewards / (rewards.std(dim=-1, keepdim=True) + 1e-6)

# loss.py::compute_advantages_and_returns
returns = get_grpo_returns(rewards, kl)  # 把每条 response 的 scalar reward broadcast 到 token
advantages = [r for r in returns]

一个例子：n_samples_per_prompt=4，同一题四条回答 raw reward 为 [1, 1, 0, 0]。组均值是 0.5，归一化前 reward 变成 [+0.5, +0.5, -0.5, -0.5]。训练时两条正确回答的 token 被整体增强，两条错误回答的 token 被整体压低。

GRPO 的优点是省掉 critic，资源和流程都简单；代价是它强依赖组内 reward 有差异。如果一组全对或全错，组内标准差为 0，训练信号就很弱，所以 Vime 有 zero_std/count_* 和 dynamic sampling filter 来监控/过滤这类组。

8.3 PPO 怎么运作

PPO 需要 critic。Vime 在 parse_args 里把 args.use_critic = args.advantage_estimator == "ppo"，所以 train.py 会先训 critic，再把 critic 返回的 values 交给 actor。

# train.py
value_refs = critic_model.async_train(rollout_id, rollout_data_ref)
ray.get(actor_model.async_train(rollout_id, rollout_data_ref, external_data=value_refs))

# critic: values -> advantages/returns -> value_loss
rollout_data.update(forward_only(get_values, ...))
compute_advantages_and_returns(args, rollout_data)
args.loss_type = "value_loss"
train(...)

# actor: 接收 critic values -> policy_loss
rollout_data["values"] = tensors_to_gpu(external_data["values"])
compute_advantages_and_returns(args, rollout_data)
train(...)

PPO 的 reward 不是简单 broadcast。Vime 会把 KL penalty 当成 token-level reward，再把最终 scalar reward 加到最后一个有效 response token 上：

# loss.py::compute_advantages_and_returns, advantage_estimator == "ppo"
for reward, k in zip(old_rewards, kl):
    k *= -args.kl_coef              # 每个 token 的 KL 惩罚
    k[-1] += reward                 # 最后一个 token 加最终任务 reward
    rewards.append(k)
advantages, returns = get_advantages_and_returns_batch(
    total_lengths, response_lengths, values, rewards, gamma, lambd
)

然后 PPO 用 GAE 算 advantage。直觉是：如果 critic 认为某个 token 之后的期望回报很高，但实际回报更高，那么 advantage 为正；实际更低则为负。actor 用 clipped policy loss 更新，critic 用 value loss 学会更准地预测 return。

8.4 policy loss 里的 ratio、KL、entropy

Vime 里的 PPO-style policy loss 统一走 ppo_utils.compute_policy_loss：

# old_log_probs 通常来自 rollout_log_probs 或上一次 actor forward
ppo_kl = old_log_probs - log_probs
ratio = exp(-ppo_kl) = exp(log_probs - old_log_probs)
pg_loss_1 = -ratio * advantages
pg_loss_2 = -clip(ratio, 1 - eps_clip, 1 + eps_clip_high) * advantages
pg_loss = max(pg_loss_1, pg_loss_2)

这个 clip 的意义是限制 actor 一次更新不要把某些 token 的概率改得太猛。entropy 则衡量分布有多“散”：太低说明模型过早变得确定，探索减少；太高说明模型可能还不稳定。kl 衡量 actor 和 reference 或 old policy 的偏离，过大通常表示训练步太激进。

8.5 怎么判断训练效果

指标从哪里来怎么看
eval/<dataset>_log_eval_rollout_data最重要。独立评估集平均 reward / accuracy 应该上升。训练 reward 上升但 eval 不升，通常是过拟合、reward hacking 或采样分布变了。
rollout/response_len/*compute_metrics_from_samples长度变化要结合任务看。数学 RL 常见模型学会更长推理，但如果暴涨且 reward 不涨，可能在拖长输出。
rollout/truncated_ratioSample.Status.TRUNCATED高截断率表示 rollout_max_response_len 不够，reward 和 loss 都可能偏。
pass@k / log_passratecompute_pass_rate对每题多采样时很有用。平均 reward 上升但 pass@k 不升，可能只是少数 easy case 变好。
rollout/zero_std/count_*_compute_zero_std_metricsGRPO 关键指标。全对/全错组越多，组内相对信号越少；必要时用 dynamic sampling 过滤。
rollout/log_probs、rollout/ref_log_probs、rollout/klMegatron forward + compute_advantages_and_returnsKL 持续走高说明 actor 偏离参考模型。太低且 reward 不变，可能更新太保守。
rollout/entropyget_log_probs_and_entropy观察探索程度。快速坍缩到很低时要检查学习率、clip、KL、reward 尺度。
rollout/pg_clipfracpolicy_loss_functionPPO clip 触发比例。长期很高通常说明 step 太大；长期接近 0 且 reward 不动，可能更新太弱。
rollout/value_loss、value_clipfracvalue_loss_functionPPO 才看。critic 学不好时 advantage 噪声大，actor 更新会不稳。
tis、tis_clipfrac、tis_absvanilla_tis_function训练策略和 rollout 策略 mismatch 的信号。越大说明当前 actor 对旧 rollout 数据的概率偏移越大。
perf/tokens_per_gpu_per_sec、rollout_timecompute_perf_metrics_from_samples性能指标。训练效果没问题但吞吐差时看这里和 dynamic batch size。
实战里优先级通常是：先看 eval reward/pass@k，再看训练 rollout reward 是否同步变化；然后检查 truncated_ratio、response_len、KL/entropy/clipfrac 是否异常。单看训练 loss 不可靠，因为 RL 的 loss 不是监督学习里的“越低越好”。

9. R3 / router replay：MoE 训练里的路由重放

R3 解决的是 MoE RL 中一个很具体的问题：rollout 时 vLLM 选择了一组 experts 来生成 token；训练时 Megatron 重新 forward 同一批 token，router 可能选出另一组 experts。这样 actor 训练的 logprob、梯度路径和真正生成 response 的路径不一致，会放大 train/infer mismatch。

9.1 MoE 里为什么会出现这个问题

MoE 层不是所有 FFN 都参与计算。每个 token 先经过 router 得到 top-k expert id，再只送到这些 experts。以 DeepSeek/Qwen MoE 这类模型为例，一个 token 在某层可能选 [17, 93, 104, ...]，下一层又是另一组。

hidden_states --router logits--> topk_ids/topk_weights
             --dispatch to selected experts-->
             expert FFN output --combine--> next hidden_states

RL 训练里，rollout 和 train 常不在同一个引擎里：rollout 是 vLLM，train 是 Megatron。即使权重刚同步过，下面这些因素也可能让 router 结果不同：kernel 数值差异、TP/EP/sequence parallel 排布差异、EPLB logical/physical expert 映射、训练 forward 与推理 forward 的微小实现差异，以及 actor 更新后旧数据重算时策略已经变了。

解析 vLLM 返回vllm_rollout._apply_vllm_routed_experts把 choices[0].routed_experts 从 base64 npy 解码成 ndarray，并检查 shape 是 (len(tokens)-1, num_layers, topk)。
进入训练数据RolloutManager._convert_samples_to_train_data强制每个 sample 都有 rollout_routed_experts，否则报错。
填充 replay bufferMegatronTrainRayActor.fill_routing_replay按 micro-batch、CP/TP/PP/VPP 排布把每层 top-k expert id 录入 RoutingReplay。
重放 routerslime/utils/routing_replay.py训练 forward/backward 时不再用当前 top-k 结果，而是 pop rollout 阶段记录的 top-k ids。

9.3 Megatron 训练时怎么重放

RoutingReplay 的核心很小：record 时把 top-k expert id 放到 CPU pinned memory；replay 时按 forward/backward 顺序取回来放到当前 GPU。

# slime/utils/routing_replay.py
if ROUTING_REPLAY_STAGE == "record":
    probs, top_indices = old_compute_topk(scores, topk, ...)
    ROUTING_REPLAY.record(top_indices)
elif ROUTING_REPLAY_STAGE in ["replay_forward", "replay_backward"]:
    top_indices = ROUTING_REPLAY.pop_forward()  # 或 pop_backward
    probs = scores.gather(1, top_indices)

注意这里不是完全绕过 router。Megatron 仍然计算当前 router 的 scores，只是 expert id 使用 rollout 的 top_indices。然后用 scores.gather(1, top_indices) 取当前模型对这些 experts 的权重。这样梯度仍然能回到当前 router logits，但专家选择路径和 rollout 对齐。

9.4 为什么要按 layer / token 对齐

vLLM 返回的 routed experts shape 是 [len(tokens)-1, num_layers, topk]。这里是 len(tokens)-1，因为对自回归模型来说，第 i 个 forward 位置产生的是下一个 token 的 logits；最后一个刚采样出来的 token 还没有被模型 forward 过。Vime 在 _apply_vllm_routed_experts 里用 expected_rows = len(sample.tokens) - 1 严格校验。

Megatron 侧还要处理 CP 和 sequence packing：训练 batch 可能把多个样本拼成一个 thd token stream，也可能把长序列切到多个 CP rank。fill_routing_replay 会先把每个 sample 的 routed experts pad 到与 token 对齐，再调用 slice_with_cp，最后按 Megatron 实际构建的 layer 顺序写入每个 MoE 层对应的 replay buffer。

9.5 vLLM 侧是怎么做的

vLLM 的开关是 --enable-return-routed-experts，对应 ModelConfig.enable_return_routed_experts。打开后，GPU worker 初始化 RoutedExpertsCapturer，并把 capture callback 绑定到每个 FusedMoE 的 router 上。

# vllm/v1/worker/gpu_model_runner.py
for module in static_forward_context.values():
    if isinstance(module, FusedMoE) and isinstance(module.router, BaseRouter):
        module.router.set_capture_fn(lambda topk_ids: capturer.capture(layer_id, topk_ids))

# vllm/model_executor/layers/fused_moe/router/base_router.py
topk_weights, topk_ids = self._compute_routing(...)
if self.capture_fn is not None:
    self.capture_fn(topk_ids)        # 捕获 logical ids，发生在 EPLB mapping 之前
topk_ids = self._apply_eplb_mapping(topk_ids)

这里“捕获 logical ids”很重要。vLLM 如果启用 EPLB，实际执行 expert 可能是 physical expert slot；但训练重放需要的是模型语义上的 logical expert id，否则 Megatron 侧没有同一个 physical slot 映射。

vLLM 内部不是把每个请求的 routed experts 直接挂在 request 上，而是先按 KV cache slot 存：

  • RoutedExpertsCapturer.device_buffer：GPU 上的 step buffer，shape 为 [max_num_batched_tokens, num_layers, topk]。
  • GPUModelRunner：每步把 device buffer 和 attention slot_mapping 拷到 CPU。
  • Scheduler.RoutedExpertsManager：按 physical KV slot 写入 routed_experts_by_slot，完成/停止请求时用 block ids 还原该请求的 token 顺序。
  • OpenAI / disagg serving：把 ndarray 保存成 .npy 字节流，再 base64 放进响应的 routed_experts 字段。
vLLM 当前实现对 R3 有约束：enable_return_routed_experts 不支持 DCP/PCP 这类 context parallel；slot-indexed CPU buffer 也会增加内存占用。Vime 文档里说 R3 依赖 vLLM 0.22+ 的 --enable-return-routed-experts 返回契约。
