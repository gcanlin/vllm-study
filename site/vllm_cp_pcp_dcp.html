vLLM Context Parallel：PCP 与 DCP 究竟怎么切？

本文围绕 CP-Viz 里的图，把 PCP（Prefill Context Parallel） 与 DCP（Decode Context Parallel） 在 vLLM 里到底是怎么对张量切的、每一步通信前后的 shape 是什么、以及 KV cache 在每个 rank 上长什么样讲清楚。所有 shape 都尽量标到了 attention head 维度上。

TL;DR
  • PCP：prefill 时把序列维度切到多个 rank 上算，每个 rank 只算 S/pcp 个 query 的 attention（采用 DualChunkSwap 让因果掩码下负载平衡），主要目标是降低 TTFT。
  • DCP：在已有 TP 切头的基础上，把同一头的 KV cache 沿 token 维度切到 dcp_size 个 rank 上，每张卡只存 1/dcp 的 KV，主要目标是把单卡 KV 扩到 1M+ token、提升吞吐与 TPOT。
  • 两者正交：PCP × DCP 既不重复物理 GPU，也不互相挤占头维，可以叠加使用。

1. 背景：从 TP 到 CP，到底为什么要切序列

先回忆一下 vLLM 已有的并行维度，attention 这一层会有三种切分方式同时存在：

维度沿什么轴切每个 rank 上 attention 实际算什么解决什么
TP tensor parallelhead 数 H本卡 H/tp 个 head，全部 token，全部上下文切权重 + 切激活的 head 维，省显存与算力
PCP prefill CPprefill 的 query token 数 S本卡 S/pcp 个 query，仍是 H/tp 个 head；KV 通过 ring/allgather 在 pcp 内流转把超长 prompt 的 TTFT 拉低
DCP decode CPKV cache 的 token 维 N本卡持有 N/dcp 个 token 的 KV，对 query 做局部 attention，再跨 rank 合并 LSE把单 TP 组的 KV 容量扩 dcp 倍；长序列 decode 时分摊 attention 计算

所以从张量切分上看：

TP 切 head 维 PCP 切 prefill query 维 DCP 切 KV cache 的 token 维（在 TP 之内继续细分）

读这张图的关键：TP / PCP / DCP 切的是三个不同维度，所以它们在一台机器上是同时叠加的。下面我们就先看物理 rank 怎么排布，再分别看两种 CP 怎么切。

2. Rank 布局与三个通信组

vLLM 把所有 worker rank 按 (ep, dp, pp, pcp, tp) 5 维数组排布，最里面变化最快的是 TP。从这块儿派生三个通信组：

参考：vllm/distributed/parallel_state.py 中 initialize_model_parallel 的 all_ranks.reshape(...)
# 简化（忽略 ep/dp/pp=1）
all_ranks = torch.arange(world).reshape(1, 1, 1, pcp_size, tp_size)

# TP group：最里面那一维
tp_groups  = all_ranks.view(-1, tp_size).unbind(0)

# DCP group：dcp_size 必须整除 tp_size，DCP 把 tp 组再切几份

# PCP group：先把 pcp/tp 转置，再按 pcp_size 划组 → 同一 tp 位置在不同 pcp slice 上的 rank 组成一组
pcp_groups = all_ranks.transpose(3, 4).reshape(-1, pcp_size).unbind(0)

为什么 DCP 必须满足 dcp_size | tp_size？ DCP 不申请新的 GPU，它复用 TP 组里现成的 rank：把原来同一个 TP 组里 tp_size 张卡，再细分成 tp_size / dcp_size 个小子组。每个小子组内的 rank 持有相同的 head 切片，但各自存一段不重叠的 KV 序列。这就是为什么我们经常说「DCP 是从 TP 里偷出来的」。

2.5.1 rank、world 和三个集合通信原语

一台机器上插了好几张 GPU。每张 GPU 在 vLLM 里叫一个 rank（编号 0、1、2 …），所有 rank 加起来叫一个 world。

关键事实是：不同 rank 的 GPU 显存是相互不可见的。rank 0 想看到 rank 1 上的一个张量，必须显式发起一次「集合通信」，让数据在 GPU 之间用 NVLink/网络真实地搬一次。

下面三个集合通信原语，是后面所有 DCP/TP 通信的基础。请你把这三张图记住，后文出现 AllGather、ReduceScatter、AllReduce 时直接对应回来：

三句话总结：
  • AllGather：拼图。每人一块小拼图，集体之后每人都拿到完整大图。张量变大（每卡 size × N）。
  • ReduceScatter：先逐位置加，再每人只拿走自己负责的那一段。张量变小（每卡 size ÷ N），但里面是「sum」。
  • AllReduce：所有卡的整张张量相加，加完之后每人各持一份完整结果。张量大小不变，但里面是「sum」。

2.5.2 TP 是怎么切 attention 这层线性的

attention 一层里有两类线性变换：

  • q_proj / k_proj / v_proj：把 hidden_states (B, hidden) 投影出 query / key / value，每个的输出形状是 (B, H × D)，H 是 head 数，D 是 head_size。
  • o_proj：把 attention 输出 (B, H × D) 投回 (B, hidden)。

TP=4 时，模型把这两个线性的权重矩阵切到 4 张卡上，但切的方向相反：

核心观察：column-parallel 的 q_proj 配上 row-parallel 的 o_proj 是一对默契组合。中间的 attention 计算只要每张卡能用自己那 H/4 个 head 算出 attention 输出就够了，不需要任何额外通信。

DCP 改变的就是这个假设：DCP 把 KV 沿 token 维进一步切了，所以「attention 计算」这一步本卡的 KV 不再完整 → 必须新增几次通信来凑。

3. DCP 详解：KV cache 怎么切，attention 怎么拼

3.1 KV cache 的 interleave 布局

DCP 的核心数据结构变化就一句话：

对绝对位置 i 的 token，它的 KV 被写到 rank (i // I) % dcp_size 上，其中 I = cp_kv_cache_interleave_size。
参考：vllm/v1/worker/gpu/block_table.py:_compute_slot_mappings_kernel
# Triton kernel 真正在做的事，简化为 Python：
for pos in positions:
    block_index   = pos // (block_size * cp_size)
    block_offset  = pos %  (block_size * cp_size)
    is_local      = (block_offset // CP_INTERLEAVE) % cp_size == cp_rank
    rounds        = block_offset //  (CP_INTERLEAVE * cp_size)
    remainder     = block_offset %    CP_INTERLEAVE
    local_offset  = rounds * CP_INTERLEAVE + remainder
    slot_id       = block_table[req][block_index] * block_size + local_offset
    slot_mapping[pos] = slot_id if is_local else PAD_SLOT_ID

其中 num_blocks 大约只需要原来非 DCP 时的 1/dcp_size，这是 DCP 能把可用 KV 容量扩到 dcp 倍的根本原因。FullAttentionSpec.max_memory_usage_bytes 里也直接把 max_model_len /= dcp_size * pcp_size 算容量。

3.2 Decode 时一层 attention 的完整通信流程

下面用最常见的 PCP=1, TP=4, DCP=4 走一遍。也就是说一共 4 张 GPU，TP 把 head 切成 4 份，DCP 把 KV cache 沿 token 维度也切成 4 份，DCP 组就是这 4 张卡本身。

场景设定（具体数字，方便对照 shape）：

  • 有 1 个请求正在 decode，所以这一步要算 B = 1 个新 token；
  • 这个请求历史已经累积了 N = 16 个 token；
  • 模型一共 H = 8 个 query head，head_size D = 128；为了简化用 MHA（H_kv = H_q = 8）；
  • 所以每张卡分到的 head 是 H/tp = 2 个。

这一步开始之前，4 张卡上各有什么？

这是最重要的一张画面。先把它在脑子里建好，后面 5 步就只是「这张画面上的格子怎么挪」而已。

请仔细看这张图，把两件事记牢：

  1. 历史 KV 已经散开了。当时往 KV cache 里写的时候（在每一层 prefill 完之后），就是按 (pos // I) % dcp 这个规则把每个 token 的 K、V 投到对应的 rank 上。pos=0 写到 rank 0，pos=1 写到 rank 1，pos=2 写到 rank 2，pos=3 写到 rank 3，pos=4 又回到 rank 0…… 一直到 pos=15。所以 rank 0 上现在只有 pos=0,4,8,12 这 4 个 token 的 KV。这就是「本卡 KV cache 里只放了 1/dcp 的历史 KV」的画面。
  2. 正确的 attention 需要 Q 的全部 8 个 head 去 attend 全部 16 个 token 的 KV。但每张卡上：head 只有 2 个，KV 只有 4 个 token。两个维度都不完整 → 需要通信去凑。

下面 5 步就是把这两件事补齐的过程。每一步我会用一段大白话讲清楚「这一步为什么需要」「这一步本卡的张量变成了什么样」。

步骤 ①：本地算出本层的新 Q/K/V，把新 K/V 写到本卡 KV cache

这一步纯本地，每张卡各自跑各自的，没有通信。

当 decode 进入这一层时，每张卡都拿到上一层的输出 hidden_states，并且各张卡上的这一份 hidden_states 是完全相同的（前一层最后做过 TP AllReduce，所以大家在同一起点）。

这里 q_proj 是 column-parallel linear（看 §2.5.2 那张图）：每张卡上只持有 W_q 的一段列，于是算出来的 Q 也只有 2 个 head：

所有卡：hidden_states (B=1, hidden)
q_proj (column-parallel)
→
rank 0: Q (1, h₀h₁, D)
rank 1: Q (1, h₂h₃, D)
rank 2: Q (1, h₄h₅, D)
rank 3: Q (1, h₆h₇, D)

K/V 同理，每张卡也只算出自己那 2 个 head 的 K/V，shape (1, 2, D)。把它们写入本地 KV cache：因为新 token 的位置是 pos = 16，而 16 % 4 = 0，所以这次写入只发生在 rank 0。其他三张卡这一层的 KV cache 这一步什么也没新增。

这就是「本卡 KV cache 里只放了当前 token 的 KV」那句话的意思：
  • 历史的 16 个 token 的 KV，是过去 16 个 decode 步「分别在 4 张卡上」一点点累积起来的（pos=0 落 rank 0，pos=1 落 rank 1，pos=2 落 rank 2，pos=3 落 rank 3，pos=4 回到 rank 0……）；
  • 当前这一步新写的 pos=16 也只落到了 1 张卡上；
  • 没有任何一张卡持有完整 16+1 个 token 的 KV。
所以下一步要算 attention，必然得让 4 张卡互相配合。

步骤 ②：对 Q 做一次 DCP AllGather，让每张卡都拿到完整 8 个 head 的 Q

为什么要凑齐 head？因为「步骤 ③ 算 attention」时，每张卡只对自己持有的那段 KV 负责，但 attention 的输出必须涵盖全部 head——少一个 head，partial 结果就没法合并。

这正是 §2.5.1 那张「AllGather」图的具体应用：4 张卡每张持有 Q (1, 2, D)，做完 AllGather 之后每张卡都得到 Q_full (1, 8, D)，即包含全部 8 个 head。

代码：vllm/v1/attention/backends/flash_attn.py:907
query_across_dcp = get_dcp_group().all_gather(query, dim=1)

步骤 ③：每张卡用「完整 Q + 自己那段 KV」算一次 FlashAttention

这一步每张卡都是纯本地计算。算完之后每张卡得到「部分 attention 输出」——之所以是「部分」，是因为本卡只看到了 4 个 KV token，剩下 12 个 token 还在别人那里。

具体到 rank 0：

→

这里多出来一个 part_lse，需要解释一下：

LSE（log-sum-exp）是什么、为什么需要？
softmax 算 attention 时其实是 out = Σᵢ softmax(qkᵢ) · vᵢ。当我们只看一部分 KV token 时，得到的 out 是「错的」——因为分母（softmax 的归一化项）只统计了本卡的 token，少算了别人那 12 个。

FlashAttention 在返回 part_out 之外，额外返回每个 head 的 part_lse = log(Σᵢ exp(qkᵢ))，即 softmax 分母的对数。这是「本卡漏算了多少」的精确信息。下一步合并的时候，4 张卡只要把各自的 LSE 互相对比一下，就可以把 4 份 part_out 按正确权重加起来得到全局正确的 attention 输出。

你不用记 LSE 公式，只要记住：part_out + part_lse 一对儿，是「我对自己那段 KV 算了 attention，但还差和其他卡合并」的标记。

步骤 ④：跨 4 张卡合并 part_out + part_lse → 拼出最终 attention 输出（再切回 H/tp）

这一步对应代码里的 cp_lse_ag_out_rs，名字本身就是「AllGather LSE → 修正 part_out → ReduceScatter」三步缩写。
代码：vllm/v1/attention/ops/common.py:cp_lse_ag_out_rs()

关键的妙处在 ReduceScatter：

  • "Reduce"（相加）：把 4 张卡的「校正后 part_out」逐位置加起来，正好得到全局 attention 输出。
  • "Scatter"（按 rank 切走自己那段）：相加的结果其实有 (1, 8, D)，但每张卡只需要自己负责的 2 个 head——所以直接切走，节省 75% 带宽和显存。

做完这一步，每张卡的 attention 输出又回到了 (1, H/tp=2, D)。注意这正好是纯 TP（不开 DCP）时 attention 输出的 shape——也就是说 DCP 的「侵入」到这里就结束了，后面 o_proj 完全不知道发生过任何事。

步骤 ⑤：o_proj + 一次 TP AllReduce → 这一层 attention 输出

这一步是标准的 row-parallel linear（看 §2.5.2 那张图），跟不开 DCP 时完全一样。
每卡：out_local (1, H/tp=2, D)
o_proj (row-parallel)
→
每卡：partial_hidden (1, hidden)
TP AllReduce
→
每卡：hidden_out (1, hidden)（完全相同的完整输出）

到这里这一层 attention 就结束了，下一层又会以「每张卡同一份 hidden_states」为起点开始。

这一层 attention 多了几次通信？ 比起纯 TP，DCP 多了三次集合通信：① AllGather(Q)、② AllGather(LSE)、③ ReduceScatter(out)。但作为回报，本卡 FlashAttention 实际算的是 1/dcp = 1/4 的 KV 序列——序列越长，DCP 越值。这也是为什么 DCP 在 128K+ 超长序列下能显著降低 TPOT。

3.4 PD 分离：prefill rank ↔ decode rank 的配对

在 PD（Prefill/Decode）分离部署里，prefill 实例和 decode 实例往往配置不一样（比如 tp_P=8, dcp_P=4 配 tp_D=4, dcp_D=2）。一个 prefill rank 算完 KV 之后，需要把自己持有的那一段 KV 切片精准送给 decode 实例里对应的 rank。

判断两个 rank 是否「覆盖同一段 KV」的条件可以拆成两条：

维度判断条件直觉
Head（TP 切）两边 KV head 组在 [0, H_kv) 上有交集P 端的 tp_rank_P 持有的 KV head 区间，要跟 D 端的 tp_rank_D 持有的区间重叠
Token（DCP 切）flat_idx_P % g == flat_idx_D % g，其中 g = gcd(dcp_P, dcp_D)，flat_idx = tp_rank·pcp_size + pcp_ranktoken 是按「(pos // I) % cp_size」分给某 rank 的；两边 cp_size 不一致时，能命中的 rank 对要满足同余

当 dcp_P == dcp_D 时，token 条件退化成 dcp_local_P == dcp_local_D，即 1:1 配对；dcp_P > dcp_D 是 N:1（多个 P rank 配同一个 D rank），反之 1:N。文档里也给了一份 numpy 的向量化匹配实现 get_prefill_ranks_for_decode_rank()，思路就是把 head 范围用 arange 算出来，再对 token 维做线性同余的 arange 枚举。

4. PCP 详解：DualChunkSwap 是怎么切的

4.1 序列切分：2 × pcp 个 chunk 的 head/tail 交错

因果 attention 下，如果按朴素方式把 prefill 的 query token 平均切给 pcp_size 个 rank，每个 rank 上 attention 的「待 attend 的 KV 长度」差异巨大：rank 0 只需要 attend 到自己那段，rank pcp-1 几乎需要 attend 整条序列。所以 PCP 采用了一个非常巧妙的 DualChunkSwap（双块换位）切法。

参考：vllm_ascend/worker/pcp_utils.py:PCPManager.update_tokens_for_pcp()

规则（设 P = pcp_size，一个请求的 padded 长度为 L）：

  1. 把 L 向上 padding 到 2P 的倍数，等分成 2P 个 chunk，标号 0..2P-1。
  2. rank r 持有 chunk r（head）+ chunk 2P - 1 - r（tail），并将 head 拼在前，tail 拼在后。

下面用一个 P=2、序列长 18 的例子直接画出来。padding 到 8 的倍数 → 24，chunk size = 3：

当 P=4 时类似，chunk 数变为 8：rank 0 拿 {0, 7}、rank 1 拿 {1, 6}、rank 2 拿 {2, 5}、rank 3 拿 {3, 4}。每对 head+tail 的 chunk 在因果掩码下「能 attend 的总 KV 长度恰好都接近 L/2」，从而保证了 prefill 时每张卡 attention 的算力负载差异极小。

4.2 PCPManager 输出的几个关键 index

切分本身只是第一步，模型 forward 时还要能在多个 rank 间互相 allgather 出完整的 hidden_states / KV，并恢复正确顺序。PCPManager 维护下面几张索引表：

名字shape作用
pcp_tokens(num_reqs,)每个请求落在本 rank 上的 token 数（含 padding）
num_pcp_pads_cpu(num_reqs,)本请求在 padding 时塞了多少个 PAD
pcp_unpad_mask_cpu(total_padded_tokens,)布尔 mask，True 表示这个位置不是 padding，做 allgather 之后用来过滤
pcp_allgather_restore_idx(total_padded_tokens,)把 allgather 出来的 [rank0_chunks, rank1_chunks, ...] 重排回原始 token 顺序的 argsort 索引
q_head_idx_tensor / q_tail_idx_tensor(num_local_query,) 各半把本 rank 的 query 拆成「head chunk」「tail chunk」两段，分别和不同 KV 段算 attention
kv_with_q_head_nomask_idx / kv_with_q_head_mask_idx(...)head chunk 对应需要 attend 的 KV：左边是已经被前面 chunk 完全覆盖、不再需要 causal mask 的部分；右边是当前 rank chunk 自己内部、还需要因果 mask 的部分

文档里有一段很形象的例子：tokens=[1, 5, 8], pcp_world_size=2，

# pcp_rank=0:  pcp_tokens=[1,4,4], positions=[0, 0,1,6,7, 0,1,6,7]
# pcp_rank=1:  pcp_tokens=[1,4,4], positions=[0, 2,3,4,5, 2,3,4,5]
# num_pcp_pads_cpu = [1, 3, 0]    # 第一条 decode 复制 1 次；第二条补了 3 个 PAD；第三条正好整除
# pcp_unpad_mask_cpu = [T, F, T,T,T,T,T, F,F, F,T,T,T,T,T,T,T,T]
# pcp_allgather_restore_idx = [0,9, 1,2,10,11,12,13, 3,4,5,6,14,15,16,17, 7,8]

注意 decode 请求在 PCP 下不被切，而是被「复制」到每个 rank 上（因为只有 1 个 token，切了反而吃亏；rank 0 当真正的 owner，其他 rank 只是配合做 allgather）。

4.3 PCP Prefill 的 ring 通信与 head/tail attention 拆分

切完之后，每个 rank 本地就有 head chunk 的 Q/K/V 和 tail chunk 的 Q/K/V。要算 attention，本卡的 Q 必须能 attend 到「全局所有 chunk 上的 KV」——这就需要 ring/allgather 通信。一个 rank 上 attention 的拆解大致如下（以 P=4、本 rank=1，所以 head chunk id=1、tail chunk id=6 为例）：

这正是「head + tail」之所以平衡的关键：

  • rank 0 持有的 chunk 是 {0, 7} → attend (0.5 + 7.5) = 8 个 chunk 的 KV
  • rank 1 持有 {1, 6} → attend (1.5 + 6.5) = 8 个 chunk
  • rank 2 持有 {2, 5} → attend (2.5 + 5.5) = 8 个 chunk
  • rank 3 持有 {3, 4} → attend (3.5 + 4.5) = 8 个 chunk

所有 rank attention 总工作量完全相同。每个 rank 实际还会把 对每个其他 chunk 的 KV 通过 ring/allgather 拿到，然后调一次 FA。具体到代码里，kv_with_q_head_nomask_idx / kv_with_q_head_mask_idx（以及 tail 对应的两个）就是把这些 KV 段在 allgathered 大张量里的下标记下来。

4.4 PCP Decode：query 复制 + KV 全副本

Decode 时序列上没有 prefill 那么长的 query，PCP 退化为：所有 rank 共享同一份 decode token（rank 0 是 owner，其他 rank 拷贝）。每个 rank 的 K/V 因为之前 prefill 都已经过 allgather + restore，每个 rank 都持有完整的 KV cache 切片（PCP 不进一步切 KV，只有 DCP 才切 KV）。

所以 PCP decode 时和不开 PCP 的 decode 几乎完全一样，只是在 logits sample 阶段会把每个 PCP rank 上「重复的 decode 输出」筛回 rank 0。这也是 PCP 在 decode 上没什么开销的原因，PCP 主要拼 TTFT。

把两者放一起，物理 rank 排布仍然按 (pcp, tp)。DCP 在 TP 内继续切；PCP 跨 TP 组形成另一层组。还以 PCP=2, TP=4, DCP=4 为例（8 张卡）：

三种通信组怎么用

  • TP all-reduce 用于 o_proj、mlp.down_proj，仍在 [0,1,2,3]/[4,5,6,7] 上做
  • DCP allgather Q + reduce_scatter out 用于 attention，作用范围与 TP group 重合（DCP 组 = TP 组）
  • PCP allgather hidden_states 用于 prefill 阶段把切开的 query 重组回完整序列，跨 [0,4]/[1,5]/[2,6]/[3,7] 这 4 组

6. 优化方向：TPA & 动态 CP

6.1 TPA（Tensor-Parallel-size-Attention）RFC #34444

注意上面 DCP decode 那张图里，步骤 ② 的 DCP all_gather(Q) 是有点浪费的：我们花了一次通信，只是为了让每张卡都拿到 H 维的 Q。TPA 的思路是：

  • 不要在 q_proj 处按 H/tp 切；改成按 H/(tp/dcp) 切（也就是 H·dcp/tp）。
  • 这样一次 q_proj 之后，每张卡天然就有 H·dcp/tp 个 head 的 Q，DCP 组里的 4 张卡合起来正好是 H。
  • 省掉一次 attention 入口处的 all_gather。

标准 TP + DCP 路径

q_proj (slice by tp)
  → Q (S, H/tp, D)
  → DCP all_gather (dim=1)     // 多一次通信
  → Q_full (S, H, D)
  → attention + cp_lse_ag_out_rs
  → out (S, H/tp, D)
  → o_proj (slice by tp)

TPA 优化路径

q_proj (slice by tp/dcp)
  → Q (S, H*dcp/tp, D)             // 直接到位
  → attention + cp_lse_ag_out_rs
  → out (S, H/tp, D)
  → o_proj (slice by tp)

6.2 动态 CP（CrossDPScheduler）RFC #29295

CP 不开/开多大并不是「全局静态决定」的最优解：短请求开 CP 反而劣化，长请求才需要。动态 CP 在 scheduler 层做了一件事：把多个 DP replica 聚合成一个 Domain，按请求序列长度动态选择 CP size：

  • 短序列：CP=1，走标准 DP；
  • 长序列：从预建立的多个 CP 通信组里挑一个合适的 CP size 跑；
  • 长尾请求：可以从 CP=1 迁移到 CP>1，避免单个 long tail 拖累整批。

对应实现要点：CrossDPKVCacheManager（统一管理跨 DP block pool）、list[SchedulerOutput]（异构调度）、ModelRunner 把 CP 请求重排到 batch 最前，并按需用 2D CUDA Graph。
